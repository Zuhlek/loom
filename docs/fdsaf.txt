Add `validator.md` files for the four phases that don't currently have one: design, plan, build, review.

## Context

A recent refactor (commits 7245ceb, 5140312, 373acfe) standardized every phase folder under `orchestrator/weave/phases/<name>/` to the same shape:

- `agent.md` — the phase producer (system prompt + inline RETURN schema)
- `contract.md` — orchestrator↔agent delegation contract
- `artifact.md` — required shape of the artifact(s) the phase produces
- `methods/` — phase-internal skills the agent dispatches
- `validator.md` — opt-in quality check, dispatched in a separate Task only when the user picks `Run quality check`

Only `phases/idea/validator.md` exists today (a narrowed version of the prior generic quality-check agent). The four other phases are structurally ready to receive their own validators, but the validation content hasn't been authored.

The architectural intent is: validators are **producer-owned**, dispatched by the orchestrator in a fresh Task with the just-completed phase's RETURN block and artifacts. They never modify phase artifacts. They write `quality-review.md` and report findings; the orchestrator surfaces the findings preview to the user, who then chooses Continue or Rerun.

## What each validator should probably check

Rough scope per phase (the Idea phase will sharpen these):

- **design** — required sections per `design/artifact.md`; decisions from `decisions.md` addressed (none ignored or contradicted); constraints from `idea.md` respected; open ambiguity items concrete enough that Plan can consume them; coverage of every user-facing flow from `idea.md`.
- **plan** — graph integrity (no cycles, all `blocked-by` resolve, every behavior in `design.md` has at least one task); slice quality (vertical not horizontal); required frontmatter on every `tasks/T-NNN.md`; HITL surfacing (decisions that would otherwise interrupt Build are surfaced as variants in Plan, per the autonomous-Build commitment); `tests.md` covers the key behaviors.

# Idea — phase-validators

## What we're building

Four new `validator.md` files — one each for the `design`, `plan`, `build`, and `review` phases — that bring all five `/weave` phases up to the same shape established by `phases/idea/validator.md`. Each new validator is an opt-in subagent that the orchestrator dispatches in a fresh Task when the user picks `Run quality check` at the rerun-or-continue surface, and reports findings into `quality-review.md` + three sections of `pipeline.md` so the user can make an informed rerun decision.

This is a **pure structural addition**. No new orchestrator behavior, no schema changes, no new dispatch paths. The change is: validators exist for more phases now.

## Users and value

- **User of the validator:** the human running `/weave`, who picks `Run quality check` after a phase returns and reads the findings preview to decide `Continue` vs `Rerun phase`.
- **Indirect user:** the orchestrator (`weave/SKILL.md`), which dispatches the validator in a fresh Task and surfaces its `quality-review.md` preview.
- **Value:** today, only Idea has a quality-check option. Reruns of Design/Plan/Build/Review are a coin flip — the user has no second opinion before burning tokens on a rerun. After this change, every phase has the same opt-in safety net.

## Scope

- `orchestrator/weave/phases/design/validator.md` — new file.
- `orchestrator/weave/phases/plan/validator.md` — new file.
- `orchestrator/weave/phases/build/validator.md` — new file.
- `orchestrator/weave/phases/review/validator.md` — new file (subject to Q2 below).
- `orchestrator/weave/SKILL.md` Load Order — update step 5 to mention all five phases' validators (currently mentions only Idea).
- `orchestrator/weave/contract.md` — update the "Phase validator" row in `Inputs` and the rerun-or-continue surface text so the two/three-option distinction is no longer drawn between Idea and others.
- `orchestrator/README.md` Layout table — generalize `phases/idea/validator.md` → `phases/<phase>/validator.md`.

Each new `validator.md` must:

- Match the shape of `phases/idea/validator.md`: title heading, `Reads`, `Writes`, `Checks` table, `Output: quality-review.md` template, `User-Facing Decision`, `RETURN` schema.
- Use the same RETURN schema (`phase: quality-check`, `status: passed|findings`, `recommendation: continue|rerun`, severities `blocker|major|minor|note`).
- Be self-contained — no shared helpers, no cross-references except to the phase's own `artifact.md` and (where useful) the phase's `methods/`.
- Write only `quality-review.md` and the three pipeline.md sections.

## Out of scope

### Boundaries

- **Read boundary:** each new validator may read any prior-phase artifact in the workspace (read-any, write-own per Q04). It may also read the just-completed phase's RETURN block (passed by the orchestrator).
- **Write boundary:** each new validator writes exactly two things — `quality-review.md` (workspace-scoped, overwritten per QC run, per Q06) and three named sections of `pipeline.md` (`Quality findings`, `Pending user input`, `Next valid action`).
- **Non-boundaries:** validators do NOT modify phase artifacts, do NOT re-execute phase work (no re-grilling, no re-running tests/smokes/mutations per Q01), do NOT call `AskUserQuestion` (orchestrator owns the user surface), do NOT touch `events.jsonl` or `artifacts.json` (orchestrator owns those).

## User flows

The Quality Check flow already exists for Idea. This project generalizes it to design / plan / build without changing the flow's shape.

### Quality Check flow — Design, Plan, Build (generalized)

```
1. User runs /weave; orchestrator dispatches phase agent (design|plan|build) in fresh Task.
2. Phase agent runs its work loop and returns its RETURN block.
3. Orchestrator validates RETURN against the schema in phases/<phase>/agent.md.
4. Orchestrator surfaces THREE-option AskUserQuestion:
     Continue           accept; advance to next phase
     Run quality check  dispatch phases/<phase>/validator.md
     Rerun phase        re-dispatch <phase> with prior artifacts
5. If user picks Run quality check:
   a. Orchestrator dispatches phases/<phase>/validator.md in fresh Task with the
      just-completed phase's RETURN block.
   b. Validator reads phase artifacts + sibling artifacts (idea.md, decisions.md,
      design.md, plan/tasks/, reports, logs) read-only.
   c. Validator runs its Checks table (per-phase rows; see Interfaces).
   d. Validator writes quality-review.md (overwriting any prior file) and updates
      pipeline.md sections Quality findings / Pending user input / Next valid action.
   e. Validator returns the RETURN block (phase: quality-check).
   f. Orchestrator surfaces findings preview with TWO-option AskUserQuestion:
        Continue     accept findings as known; advance
        Rerun phase  re-dispatch <phase> with prior artifacts + findings
6. If user picks Continue: advance phase, update pipeline.md, exit.
7. If user picks Rerun phase: re-dispatch same phase agent (which reads prior
   artifacts + quality-review.md if present as additional context).
```

### Review remains two-option

Review continues to surface the existing two-option `Continue | Rerun phase` decision. No `Run quality check` option is exposed for Review (Q02 resolution: Review IS the project validator; a validator-of-the-validator is a redundant regress).

### Idea — unchanged

The existing Idea + QC flow is preserved bit-for-bit. No regression. `phases/idea/validator.md` is not edited.

## Data model

### `quality-review.md`

- **Location:** `.loom/<project>/quality-review.md` — single workspace-scoped file.
- **Lifecycle:** overwritten on each QC run, by whichever validator runs (Q06). The file's `# Quality Review — <phase>` heading distinguishes which phase produced the current contents; no prior run is preserved.
- **Schema:** as shown in Interfaces §5 above.
- **Owner-on-write:** the dispatched validator only. The orchestrator never writes this file. Other validators never read it (they read upstream phase artifacts, not the QC output of other phases).

### `pipeline.md` sections updated per QC run

All three new validators update the same three sections of `.loom/<project>/pipeline.md`, mirroring how the Idea validator already behaves:

| Section | Content written by validator |
| --- | --- |
| `Quality findings` | One-line preview per finding (severity + title); `_(none)_` when status is `passed`. Single-slot; overwritten per QC run. |
| `Pending user input` | Set to a brief pointer at `quality-review.md` so the orchestrator's rerun-or-continue surface knows findings are awaiting review; cleared by the orchestrator after the user decides. |
| `Next valid action` | Set to a short directive like `Review quality-review.md, then choose Continue or Rerun phase`. |

The other `pipeline.md` sections (`Current phase`, `Phase status`, `Produced artifacts`, `Resume point`, `History`) remain owned by the orchestrator. Validators never touch them.

### RETURN YAML block

Identical to Idea — schema reproduced in Interfaces §7. Severities `blocker | major | minor | note`; recommendation `continue | rerun`; statuses `passed | findings`. The schema is restated verbatim per validator file (per Q05's restate-the-rubric pattern applied here to the RETURN block — keeps each file self-contained; no `include` mechanism is introduced).

### Severity rubric — restated per file (Q05)

Each of the three new validator files restates the same one-sentence rubric verbatim, matching the wording already in `phases/idea/validator.md`:

> Severities: `blocker`, `major`, `minor`, `note`. A `blocker` finding implies the next phase cannot consume the output; major implies a likely regression; minor / note are polish.

This is in addition to (not a replacement for) the severity enum in the RETURN YAML block.

### Error modes (validator-internal)

| Condition | Validator response |
| --- | --- |
| A required input file is missing (e.g. `test-report.md` for Build) | Treat as a finding: severity `major`, title names the missing file. The validator does not crash; it returns `status: findings` with the issue listed. |
| The phase artifact is malformed (e.g. `design.md` has no front matter) | Treat as a `blocker` finding. The validator does not attempt to re-author the artifact (writes-own constraint). |
| A check requires reading a sibling artifact that doesn't yet exist (e.g. Build validator asked to run before Plan completed) | Validator returns `status: findings` with a single `blocker`: prerequisite phase artifact missing. |

No internal `AskUserQuestion`. Anything user-facing is funnelled through `quality-review.md` and the orchestrator's two-option surface.

## Constraints

Carried forward from `idea.md` Constraints + Conventions-adopted-from-grilling:

- **Read-only model (Q01, Q04).** Validators read what they need (own-phase artifacts + sibling artifacts) and never re-execute phase work. Build's validator reads `test-report.md` / `smoke-report.md` / `tasks/T-*.test-log.txt` / `tasks/T-*.done.md`; it never re-runs smoke or mutation tests.
- **Title convention (Q03).** Each file opens with `# <Phase> Validator` — `# Design Validator`, `# Plan Validator`, `# Build Validator`. Matches the existing `# Idea Validator`.
- **Severity rubric restated per file (Q05).** Each new validator file restates the one-sentence rubric verbatim — `blocker` = next phase cannot consume the artifact; `major` = likely regression; `minor` / `note` = polish.
- **Single shared `quality-review.md` overwritten per QC run (Q06).** One workspace-scoped file at `.loom/<project>/quality-review.md`. No per-phase QC files. No append-mode history.
- **RETURN schema identical to `phases/idea/validator.md`.** No schema extension, no new enums.
- **No shared helper files.** Each `validator.md` is self-contained. The RETURN schema, the severity rubric, the User-Facing Decision paragraph, and the Output template are all restated verbatim per file (no `include` mechanism is introduced).
- **No orchestrator behavior changes.** The dispatch mechanism, fresh-Task semantics, and rerun-or-continue logic stay as they are. The three surface edits to `SKILL.md` / `contract.md` / `README.md` are documentation-only — they describe the new reach of the existing mechanism, they do not change it.
- **No phase artifact modifications.** Validators only write `quality-review.md` and three pipeline.md sections.
- **No regression in existing Idea + QC flow.** `phases/idea/validator.md` is not edited. The Idea phase's three-option surface is preserved bit-for-bit.

## Alternatives considered

Rejected during Idea grilling — recorded here for design provenance, not re-debated.

- **Per-phase `quality-review-<phase>.md` files** (Q06 No). Would preserve prior QC outputs across phase transitions but expands the orchestrator contract — the `Quality findings` / `Pending user input` / `Next valid action` sections of `pipeline.md` are single-slot today and the orchestrator surfaces a single `quality-review.md`. Rejected to keep the orchestrator contract untouched (in scope of the "no orchestrator behavior change" constraint).

- **Strictly producer-owned reads** (Q04 No). Would lock each validator to reading only its own phase's artifacts. Would make many of the seed-named checks un-implementable: Design's "constraints from idea.md respected", Plan's "every behavior in design.md has at least one task", Build's task-vs-spec cross-reference. Rejected because the precedent (Idea validator reading `seed.md`) already establishes read-any/write-own.

- **Build validator re-running smoke / mutation tests** (Q01 No). Would give independent verification but doubles token / time cost and risks sandbox drift between phase and validator runs. Rejected as a divergence from the read-only Idea-validator precedent and disproportionate to the failure mode (Build already runs these inline during the phase).

- **`review/validator.md` — full-scope** (Q02 A). Would check Review Targets coverage, finding shape, severity consistency, log appends, and re-judge Review's substantive findings. Rejected because Review IS the project validator; second-guessing its substantive findings is a validator-of-validator regress.

- **`review/validator.md` — narrow-scope** (Q02 B). Would limit the Review validator to artifact-shape checks only (required sections, finding-shape, log appends), without re-judging findings. Rejected because the user accepted the regress argument and chose to break the seed's "five phases have validators" framing; scope drops to 3 new files.
