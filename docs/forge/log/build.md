## 2026-05-05 — forge-ui-package — Task: T-007
**Skill:** build
**Type:** Uncategorized
**What worked:** One chokidar instance with dynamic `add(path)` / `unwatch(path)` keyed by per-project `listeners.size === 0` cleanup is the right shape — fd pressure scales with "actively viewed projects," not "configured projects." Per-path byte-offset state gives both incremental reads and free truncate detection (`stat.size < offset` → reset to 0 + emit `file-changed` envelope). Trailing-partial-line rollback (find last `\n`, byte-count the tail, decrement offset by that many UTF-8 bytes) means half-written records don't trigger spurious `parse-error` envelopes — they're just deferred to the next chokidar fire when the rest of the line lands. The `rootScans: () => scanRoots(state.resolvedRoots)` getter pattern (lifted from T-002's "live state reference" decision) keeps the watcher decoupled from config hot-reload — nothing inside the watcher knows about config plumbing. In-test SSE consumer via `fetch` + `response.body.getReader()` + manual line buffering is ~80 lines and covers `data:` / `:` / blank-line framing — zero `eventsource` dep, zero need to ship a polyfill for the test target Node version.
**What didn't:** The chokidar lazy `add`/`unwatch` dance is more subtle than "first-subscriber adds, last-unsubscriber removes" because subscription resolves the project's `events.jsonl` path *asynchronously* (via `rootScans()` which is async), so a subscriber that arrives while resolution is in flight needs to be parked in the listener Set — not dropped on the floor — and the resolution callback must check that the placeholder `ProjectEntry` it created is still the entry under that key (could've been disposed and re-created concurrently). Settled on: insert placeholder synchronously, populate `filePath`/`rootPath` + `fsWatcher.add()` after resolve, all listeners attached pre-resolve still fire post-resolve. Byte-offset tracking has one edge most "tail -f" tutorials gloss over: when `read(buffer, 0, length, offset)` reads UTF-8 mid-codepoint at the chunk boundary, naive `text.split('\n')` works (the high bytes of a multibyte sequence aren't `\n` = 0x0A) but `Buffer.byteLength(text.slice(...), 'utf8')` is the right way to roll the offset back — `text.length` would undercount by the multi-byte char widths. The Fastify `reply.raw` route signature: returning `reply` from the handler tells Fastify not to auto-send anything; *not* returning anything would leave the handler's promise unresolved and Fastify would eventually 500. The in-test SSE consumer's `waitFor(predicate, timeoutMs)` needed to handle the case where the predicate is already true at call time *and* the case where multiple records arrive in one `read()` chunk (so the consumer notifies waiters once per record-boundary, not once per chunk).
**Type knowledge:** Reusable shape for "watch append-only file + emit per-line SSE + dispose-clean": (1) one shared `chokidar.watch([])` instance with dynamic `add`/`unwatch` keyed by subscriber count, never one watcher per file; (2) per-path `{ offset, debounceTimer, listeners }` map + 50 ms debounce — chokidar's `change` fires per `write(2)`, debounce coalesces bursts and matches the user's perception of "one append"; (3) on shrink, reset offset to 0 and emit a `file-changed` envelope so clients can render a "rotated" indicator; on `unlink`+`add`, also reset offset (chokidar re-fires `add` for log-rotate); (4) trailing-partial-line rollback: find last `\n`, roll offset back by `Buffer.byteLength(tail, 'utf8')`, leave consumable = `text.slice(0, lastNewline + 1)` — never split mid-line; (5) seed offset to current file size on first attach so late connectors don't replay history; (6) `parse-error` envelopes (don't crash) on bad JSON; (7) Fastify SSE: `reply.raw.writeHead(200, headers)` + `flushHeaders?.()` + `reply.raw.write('data: ...\n\n')`; return `reply` from the handler so Fastify cedes the response; clean up via `request.raw.on('close', cleanup)`; (8) heartbeat as `: comment\n\n` (not a `data:` frame) so EventSource ignores it; 30 s is the SSE-spec-friendly default; (9) test SSE consumer: `fetch` + `body.getReader()` + TextDecoder({stream:true}) + scan for `\n`, on blank line emit accumulated `data:` lines as one record, ignore `:` heartbeats — total ~80 LOC, zero deps.

**What worked:** Splitting the new-forge bootstrap into three concern-isolated modules — `spawn.ts` (no Fastify), `seed.ts` (no Fastify, no spawn), `routes/new-project.ts` (Fastify glue) — kept each piece independently testable and let the test mock the spawn side via the existing `spawnCommand` config knob without any DI plumbing. Piping seed body via stdin to atomic-write.sh sidesteps argv length limits and shell quoting hazards entirely — exec(`bash`, [script, path]) + child.stdin.end(body) is two lines and bullet-proof. The fd-handoff dance for detached spawn (open fs.FileHandle in append, pass fd to stdio, child.unref(), then close the parent fs.FileHandle while the child keeps its own dup) is exactly the right Node idiom: the child writes for as long as it lives, the parent doesn't leak an fd. Mapping `ScaffoldError.code` 1:1 to the spec's 4xx error strings (`invalid-slug`, `unknown-root`, `project-exists`, `invalid-seed`) means the route layer is one switch and the test asserts on a single contract surface.

Flagged in done.md "Notes for downstream tasks" with a concrete remedy (config.forgeRoot). Detached spawn fd handoff has a subtle ordering trap: if you close the parent's fs.FileHandle BEFORE spawn(), the kernel may have already invalidated the fd by the time spawn() consumes it; if you call child.unref() AFTER closing the handle but BEFORE the child has actually started, you get races on slow systems. Settled order: open → spawn → unref → close, and rely on the child holding its own dup. Slug-validation duplication client+server is the classic UX-vs-DRY tension; chose duplication for the inline-validation snappiness and noted it for the design council later. Initial instinct was to validate the seed body length only on the server, but the textarea character-count UX needs the same number client-side, so MAX_SEED_BYTES is duplicated too — both are constants, not regex, so drift is harder to introduce.
**Type knowledge:** Reusable shape for "fire-and-forget detached subprocess + per-process log file + 201 response": (1) `await fs.mkdir(logDir, { recursive: true })` first — any `spawn()` of a process whose stdio fd targets a non-existent directory will silently swallow output on some platforms; (2) `const handle = await fs.open(logPath, 'a'); spawn(cmd, argv, { stdio: ['ignore', handle.fd, handle.fd], detached: true });` — append mode lets the child re-enter on log rotation safely, and one fd serves both stdout and stderr; (3) `child.unref()` + `await handle.close()` AFTER spawn returns — order matters; (4) ISO timestamps in filenames need `.replace(/[:.]/g, '-')` because `:` is forbidden on Windows-mounted filesystems and `.` confuses some `tar`-based round-trips even though it's technically legal; (5) `spawn(cmd, [...args, '/craft <name>'])` (one positional arg "with a space") matches `claude -p "/craft foo"` shell-equivalent invocation — DON'T split into `['/craft', 'foo']`, the framework's slash-command dispatcher reads the single-token form; (6) shell-out for atomic file writes from Node beats reimplementing rename(2) atomicity in JS — keeps one canonical writer (`forge/lib/atomic-write.sh`) across forge subagents and forge-ui server; (7) Best-effort emit-event: don't fail the POST when the helper script trips on a missing `jq` or quirky bash — log to stderr and continue; the 201 is about the disk state, not the event log; (8) UI optimistic-update + 500 ms re-fetch is the right cadence: prepend immediately so the user sees their action, re-fetch once to absorb whatever the spawned subprocess wrote; longer waits feel laggy, shorter waits race the subprocess's first disk write.
