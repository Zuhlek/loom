# T-005 — DELETE /api/bookmarks/:id with 404 on missing

## Goal

Add the server slice of US-004: `DELETE /api/bookmarks/:id` removes
the row when it exists and returns `404 not_found` when it does not.

## Scope

- Extend `src/server/routes.ts` with `DELETE /:id`.
- Parse `:id` as integer; on parse failure respond `400 validation
  field=id` (defensive — the path regex could also restrict it).
- Call `db.remove(id)`. If it returns `true`, respond `204` no body.
  If it returns `false`, respond `404 { error: { code: 'not_found' } }`.

## Acceptance criteria

(EARS → US-004 ACs at the server.)

1. WHEN supertest issues `DELETE /api/bookmarks/{existingId}`, the
   system SHALL respond `204` and a subsequent GET SHALL NOT include
   that row. (US-004 AC1.)
2. WHEN supertest issues `DELETE /api/bookmarks/{missingId}`, the
   system SHALL respond `404` with `body.error.code === 'not_found'`.
   (US-004 AC3.)
3. WHEN supertest issues `DELETE /api/bookmarks/abc`, the system
   SHALL respond `400` (non-numeric id).

## Behaviour-level test sketch

`test/api.test.ts` (extends T-003/T-004):

- `DELETE existing returns 204 and removes from list` — POST a row,
  capture its id, DELETE it, assert 204; subsequent GET returns
  empty array.
- `DELETE missing returns 404 not_found` — DELETE a fabricated id
  like `999`; assert status 404 and error.code.
- `DELETE non-numeric returns 400` — `DELETE /api/bookmarks/abc`;
  assert 400.
- `DELETE leaves other rows intact` — POST two rows, DELETE the
  first, GET; assert the second row is still present.
