# T-007 — Client api wrapper and dom render helpers

## Goal

Build the two client-side foundations described in `design.md
§ Client module signatures`: `api.ts` (fetch wrappers + error
normalisation) and `dom.ts` (pure render helpers). After this task,
T-008/T-009/T-010/T-011 can compose them into user-visible flows.

## Single-layer justification

`api.ts` and `dom.ts` are below the user-visible surface — by design
they have no `window` or `document` side effects beyond their inputs.
Splitting them into their own slice lets each consumer task (list,
save, open, delete) be tested against a stable API + a stable DOM
contract.

Every story is listed because removing this task would block delivery
of every client behaviour slice.

## Scope

- `src/client/api.ts`:
  - `listBookmarks(): Promise<Bookmark[]>` — `GET /api/bookmarks`.
  - `createBookmark({title, url}): Promise<Bookmark>` — `POST` JSON.
  - `deleteBookmark(id): Promise<void>` — `DELETE`.
  - On non-2xx, parse `body.error` and reject with an
    `ApiClientError extends Error` carrying `code`, `message`,
    `field?`.
- `src/client/dom.ts`:
  - `renderList(target, items)` — clears `target`, appends one `<li>`
    per item containing a title link (`<a target="_blank"
    rel="noopener noreferrer" href={item.url}>{item.title}</a>`),
    a url span, and a `<button data-action="delete"
    data-id={item.id}>` control.
  - `renderEmptyState(target)` — clears `target`, appends an empty-
    state node (or toggles the existing `#empty-state` node visible).
  - `renderFieldError(form, field, message)` — adds the error
    message under the matching field; `field === 'form'` writes to
    the form-level slot.
  - `clearFieldErrors(form)` — removes all error nodes added by
    `renderFieldError`.

## Acceptance criteria

1. `listBookmarks` resolves to the `bookmarks` array on a 200.
2. `createBookmark` resolves to the bookmark on a 201; rejects with
   `ApiClientError(code='duplicate', field='url')` on a 409;
   rejects with `ApiClientError(code='validation', field=…)` on a
   400.
3. `deleteBookmark` resolves on 204; rejects with
   `ApiClientError(code='not_found')` on a 404.
4. `renderList([])` leaves the target empty; the consumer is
   responsible for showing the empty state.
5. `renderList([item])` produces a single `<li>` with the title `<a>`
   carrying `target="_blank"` and `rel="noopener noreferrer"` and a
   delete button with `data-id` equal to the item's id.
6. `renderFieldError` is idempotent — calling it twice for the same
   field replaces the previous message rather than stacking nodes.

## Behaviour-level test sketch

`test/client/api.test.ts` (Vitest + `vi.spyOn(globalThis, 'fetch')` or
the `undici`/`msw` shim of choice):

- `listBookmarks resolves to bookmarks on 200`.
- `createBookmark rejects with ApiClientError code=duplicate on 409`.
- `createBookmark rejects with code=validation field=title on 400`.
- `deleteBookmark rejects with code=not_found on 404`.

`test/client/dom.test.ts` (Vitest `environment: 'jsdom'`):

- `renderList produces one <li> per item with target=_blank link` —
  Build a `<ul>`, call `renderList(ul, [{id:1,title:'X',url:'https://x.test',createdAt:'…'}])`,
  assert one `<li>`, one `<a target="_blank" rel="noopener
  noreferrer" href="https://x.test">X</a>`, one button with
  `data-id="1"`.
- `renderList clears the target before re-rendering`.
- `renderEmptyState toggles the empty-state node visible`.
- `renderFieldError(form, 'url', 'm') is idempotent` — call twice,
  assert only one error node under the url field.
- `clearFieldErrors removes prior error nodes`.
