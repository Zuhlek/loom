# T-008 — Render bookmarks list and empty state on page load

## Goal

Wire the first user-visible client behaviour: on page load, fetch the
bookmark list, render it newest-first, or show the empty-state node
when the list is empty. This is the client slice of US-002.

## Scope

- `src/client/main.ts` (initial form):
  - On `DOMContentLoaded`: look up `#bookmark-list` and `#empty-state`;
    call `listBookmarks()`; render the list or the empty state.
  - Export a `bootstrap(doc, deps)` factory so tests can drive the
    code without `DOMContentLoaded` timing (deps = `{ api, dom }`).
  - On fetch error: render a banner in `#banner` (non-fatal — list
    stays in its prior state).

## Acceptance criteria

(EARS → US-002 ACs.)

1. WHEN `bootstrap` runs against a DOM containing the shell and
   `listBookmarks` resolves to two items, the system SHALL render
   both items into `#bookmark-list`, newest-first. (US-002 AC1.)
2. WHEN `listBookmarks` resolves to an empty array, the system SHALL
   show `#empty-state` and SHALL leave `#bookmark-list` empty.
   (US-002 AC3.)
3. Each rendered row SHALL contain the title, the url text, and a
   delete button (delete behaviour itself lands in T-011). (US-002 AC2.)

## Behaviour-level test sketch

`test/client/main-list.test.ts` (Vitest jsdom):

- `bootstrap renders two items newest-first` — Set up a DOM with the
  shell, inject a stub `api` whose `listBookmarks` resolves
  `[{id:2,title:'B',…},{id:1,title:'A',…}]`; call `bootstrap`; assert
  `#bookmark-list` has two `<li>` in that order.
- `bootstrap shows empty state when list is empty` — Stub
  `listBookmarks` resolves `[]`; assert `#empty-state` is no longer
  `hidden` and `#bookmark-list` has zero children.
- `bootstrap shows banner on fetch error` — Stub `listBookmarks`
  rejects; assert `#banner` becomes visible with a message and
  `#bookmark-list` is untouched.
- `each row has title, url text, and a delete button` — assert child
  structure of the rendered `<li>` matches the contract from T-007.
