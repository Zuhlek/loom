# T-011 ŌĆö Delete control removes row with non-fatal 404 handling

## Goal

Wire the per-row delete button: clicking it calls `deleteBookmark`,
removes the row from the DOM on success, and on a `404 not_found`
surfaces a non-fatal banner while still removing the row from the
local list. This is the client slice of US-004.

## Scope

- `src/client/main.ts` (extended):
  - On click of `[data-action="delete"]` in `#bookmark-list`
    (event-delegated, so dynamically prepended rows are also
    handled): read `data-id`, disable the button, call
    `deleteBookmark(id)`.
  - On 204 success: remove the row from the DOM; if the list is now
    empty, show `#empty-state`.
  - On `ApiClientError(code='not_found')`: render a non-fatal banner
    in `#banner` reading "This bookmark was already removed.";
    remove the row from the DOM anyway (the local view should match
    the server).
  - On any other error: keep the row, re-enable the button, surface
    the message in `#banner`.

## Acceptance criteria

(EARS ŌåÆ US-004 ACs.)

1. WHEN the user clicks a row's delete button and
   `deleteBookmark` resolves, the system SHALL remove that row from
   `#bookmark-list` without a page reload. (US-004 AC1, AC2.)
2. WHEN removing the last row, the system SHALL show `#empty-state`.
3. WHEN `deleteBookmark` rejects with `code='not_found'`, the system
   SHALL render a non-fatal banner in `#banner` AND SHALL remove the
   row from `#bookmark-list`. (US-004 AC3.)
4. WHEN `deleteBookmark` rejects with any other error, the row SHALL
   remain in the DOM and the delete button SHALL be re-enabled.

## Behaviour-level test sketch

`test/client/main-delete.test.ts` (Vitest jsdom):

- `click delete removes the row on 204` ŌĆö Boot the page with two
  stub bookmarks; stub `deleteBookmark` to resolve; click the first
  row's delete button; assert only the second row remains.
- `deleting the last row shows empty state` ŌĆö One bookmark; click
  delete; assert `#bookmark-list` is empty and `#empty-state` is
  visible.
- `404 surfaces banner and still removes the row` ŌĆö Stub
  `deleteBookmark` to reject with `ApiClientError(code='not_found')`;
  click delete; assert `#banner` contains the "already removed"
  message and the row is gone.
- `other error keeps the row` ŌĆö Stub `deleteBookmark` rejects with a
  generic `ApiClientError`; click delete; assert the row is still in
  the DOM and the button is enabled.
- `event delegation handles a freshly prepended row` ŌĆö Prepend a new
  row via the save flow (or directly via `prependItem`); click its
  delete button; assert it is deleted. This guards against
  per-element listeners attached only at initial render.
