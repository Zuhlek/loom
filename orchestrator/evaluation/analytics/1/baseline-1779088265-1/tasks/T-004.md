# T-004 — Save a bookmark end-to-end

## Why

Vertical slice for US-001 — the user submits a URL + title and sees it added. Cuts through every layer (repo → route → client api → form → render of form error) so a single review covers the whole behaviour. Pairs with T-005 (list) so the user can see the result, but does not strictly require T-005: a POST returning `201` is observable on its own.

## Layer coverage

- `repo` — `insert(input)` returning the new `Bookmark`; `DuplicateUrlError` class implementation; mapping `SQLITE_CONSTRAINT_UNIQUE` to the typed error; rowToBookmark camelCase mapping.
- `routes` — `POST /api/bookmarks` handler: validates body (non-empty `url`, non-empty `title`, `new URL(url)` succeeds), calls `repo.insert`, returns `201 {bookmark}`; on `DuplicateUrlError` lets the error middleware return `409`.
- `client-api` — `createBookmark(input)` in `api.ts`; parses `{bookmark}`; throws `ApiError` on non-`201`.
- `client-form` — form state machine `idle → submitting → idle | error`; client-side `URL` constructor validation before fetch; surfaces server `message` into the error state.
- `client-render` — `renderFormError(root, message)` shown / cleared by `form.ts`.

## Acceptance criteria

These map 1:1 to US-001's EARS clauses.

1. **AC1 (happy path):** `POST /api/bookmarks {url: 'https://x.com', title: 'X'}` returns `201` with `{bookmark: {id, url, title, createdAt}}`; a subsequent `GET /api/bookmarks` (which lands in T-005 but is exercised here against the in-memory repo directly) includes the row.
2. **AC2 (validation):** missing or empty `url`, missing or empty `title`, or `new URL(url)` throwing → `400` with `{error: {code: 'VALIDATION', message}}`; row count in SQLite unchanged.
3. **AC3 (duplicate):** second `POST` with the same URL → `409` `{error: {code: 'DUPLICATE_URL', message: 'URL already saved'}}`; original row unchanged (same `id`, `createdAt`, `title`).
4. **AC4 (timestamp):** the inserted row's `createdAt` is `Date.now()`-shaped (positive integer; within ±2s of the test's clock at insert time).
5. **Client form state:** under jsdom, dispatching a submit event with valid inputs transitions `idle → submitting → idle`, clears the input fields, and calls `api.createBookmark` once. On `400`/`409`, transitions to `error`, displays the server `message` via `renderFormError`, does not clear the inputs.

## Behaviour-level test sketch

- `test/repo.test.ts` (insert section): insert one row → returned object matches `{id: number, url, title, createdAt: number}`; second insert with same URL → throws `DuplicateUrlError` with `code === 'DUPLICATE_URL'`; row count remains 1.
- `test/routes-create.test.ts` (supertest against `createApp({ repo: createBookmarksRepo(openDb(':memory:')), staticDir })`):
  - `POST` valid → `201` + `bookmark` shape.
  - `POST` empty title → `400 VALIDATION`; DB row count 0.
  - `POST` `url: 'not a url'` → `400 VALIDATION`.
  - `POST` valid; `POST` same url again → `409 DUPLICATE_URL`; `GET` (via direct repo) returns one row.
  - Body > 10kb → `413` (from T-003 limit).
- `test/client-form.test.ts` under jsdom: stub `api.createBookmark` with a controllable promise; assert state transitions and DOM side-effects via `renderFormError`. Confirm `new URL(input)` validation runs before `fetch` (no network call on malformed URL).

## Notes

- Time generation lives in the repo (`Date.now()` at `insert` time) — Design `§ ADR-003`. Tests pin time with `vi.setSystemTime` to assert AC4.
- Repository converts `created_at` (snake) → `createdAt` (camel) at the boundary.
- Form does NOT optimistically prepend the row; it calls into T-005's `loadAndRender()` (which lands in T-005) after `201`. For T-004 alone, the form can call a passed `onSaved` callback that T-005 wires up.
- `renderFormError(root, null)` clears the error region. No `innerHTML`.
