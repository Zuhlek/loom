# T-005 — List bookmarks newest-first end-to-end

## Why

Vertical slice for US-002 — every saved bookmark shows in one list, newest first, with an empty-state when none exist. Independently demoable: with T-004 in place a user can save, refresh, and see; without T-004 the gate is still observable by seeding the DB directly.

## Layer coverage

- `repo` — `list()` returning `Bookmark[]` ordered by `(created_at DESC, id DESC)`.
- `routes` — `GET /api/bookmarks` handler returning `200 {bookmarks: Bookmark[]}`.
- `client-api` — `listBookmarks()` in `api.ts`.
- `client-render` — `renderList(root, bookmarks)` building DOM via `createElement` + `textContent`; `renderEmptyState(root)` rendering the empty-state message.

## Acceptance criteria

These map to US-002's EARS clauses.

1. **AC1 (order):** Given three inserted rows with `created_at` `t1 < t2 < t3` (controlled via `vi.setSystemTime`), `repo.list()` returns `[row(t3), row(t2), row(t1)]`. Same-ms ties break by higher `id` first.
2. **AC2 (fields):** Each rendered list entry exposes the bookmark's `title` and `url` to the user. In the DOM, `.bookmark-title` text equals `bookmark.title`; `.bookmark-url` text equals `bookmark.url` (or the anchor's `href` — chosen at render time, both acceptable as long as it's visible).
3. **AC3 (empty state):** When the DB is empty, the client renders the empty-state message (e.g. "No bookmarks yet.") inside the list container instead of an empty `<ul>`/`<ol>`. The container is not absent — a single empty-state element occupies it.
4. **HTTP:** `GET /api/bookmarks` returns `200` with `{bookmarks: [...]}` matching the repo's order; empty DB → `{bookmarks: []}`.
5. **Client wiring:** `loadAndRender(root)` calls `api.listBookmarks()`, then either `renderList` (non-empty) or `renderEmptyState` (empty). Network failure → render an inline retry message (Design `§ Client-side state`); does not throw.

## Behaviour-level test sketch

- `test/repo.test.ts` (list section): insert three with controlled times; `list()` returns them newest-first; insert two within the same ms; both appear and the higher `id` is first.
- `test/routes-list.test.ts` (supertest): seed via repo; `GET /api/bookmarks` → `200` + ordered array; empty DB → `{bookmarks: []}`.
- `test/client-render.test.ts` under jsdom:
  - `renderList(root, [b1, b2])` → `root.querySelectorAll('[data-bookmark]').length === 2` in the order passed; each entry's title and url texts match (using `textContent`, never `innerHTML`).
  - `renderEmptyState(root)` → `root.querySelector('[data-empty-state]')` exists; no list elements present.
  - Calling `renderList(root, ...)` after `renderEmptyState(root)` (or vice versa) clears the prior content — the container has one state at a time.
  - Render of a title containing `<script>alert(1)</script>` results in the literal string in `textContent`; no `<script>` element exists in the DOM.
