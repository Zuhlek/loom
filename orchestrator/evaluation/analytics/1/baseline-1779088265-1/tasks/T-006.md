# T-006 — Open a bookmark in a new tab

## Why

Vertical slice for US-003. Tiny but distinct from T-005: list rendering owns layout and field surfacing; this task owns the anchor's exact attribute set that makes clicks open the URL in a new tab safely. Splitting it lets the security-shaped assertion (`rel="noopener noreferrer"`) live in its own commit and keeps a regression to that rule easy to spot.

## Layer coverage

- `client-render` — list-row rendering produces an `<a>` whose `href`, `target`, and `rel` exactly match the spec.
- `static` — CSS rules in `public/styles.css` so the anchor is visually click-target-shaped (block-level row, hover affordance). No JS behaviour beyond the anchor's native semantics.

## Acceptance criteria

These map to US-003's EARS clauses.

1. **AC1 (new tab):** Each rendered bookmark is an `<a>` element. Activating it (browser default behaviour, asserted via the `target` attribute) opens the URL in a new browser tab.
2. **AC2 (attributes):** Each anchor has:
   - `href === bookmark.url` (exact),
   - `target === '_blank'`,
   - `rel === 'noopener noreferrer'` (exact string, in that order or `'noreferrer noopener'` — both acceptable as long as both tokens are present).
3. No `onclick` handler simulates the new-tab behaviour — the anchor's native semantics carry it. (Avoids `window.open` and its popup-blocker pitfalls.)
4. The anchor's `href` is set via `setAttribute('href', bookmark.url)` — never via `innerHTML` string interpolation.

## Behaviour-level test sketch

Extends `test/client-render.test.ts` (the file landed in T-005):

- `renderList(root, [{id:1, url:'https://x.com/a', title:'A', createdAt: 0}])` →
  - `root.querySelector('a')` exists;
  - `.getAttribute('href') === 'https://x.com/a'`;
  - `.getAttribute('target') === '_blank'`;
  - `.getAttribute('rel')` split on whitespace contains both `noopener` and `noreferrer`.
- Render a URL containing a JavaScript-scheme-ish string (e.g. `javascript:alert(1)`) — the `href` is whatever was passed (we do not sanitise here; the validation gate in T-004 should have rejected non-`http(s)` URLs via `new URL`). This test documents the boundary: T-006 only owns the attributes, T-004 owns the URL gate.
- No element produced by render has any `onclick` listener (assert via `[...root.querySelectorAll('*')].every(el => !(el as any).onclick)`).

## Notes

- CSS adds row hover and focus styles so keyboard activation of the link is visible — supports US-003 AC1 for keyboard users without adding new behaviour.
- Per Spec `§ Constraints § Security`, the `rel="noopener noreferrer"` is non-negotiable; this task is the place where that rule lives.
