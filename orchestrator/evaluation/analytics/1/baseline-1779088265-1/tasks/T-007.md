# T-007 — Delete a bookmark end-to-end

## Why

Vertical slice for US-004 — user removes a bookmark and the list updates. Comes after both T-004 (so the same-URL-can-be-re-added clause is testable on a real save path) and T-005 (so the rendered list owns the delete control's host).

## Layer coverage

- `repo` — `deleteById(id)` returning `boolean` (true if a row was removed); never throws on missing row.
- `routes` — `DELETE /api/bookmarks/:id`: parses `id` as integer (NaN → `400 BAD_ID`); calls `deleteById`; always `204` on integer id, regardless of whether a row was removed (ADR-005).
- `client-api` — `deleteBookmark(id): Promise<void>`; throws `ApiError` on non-`204`.
- `client-render` — each list row carries a delete control (button with `aria-label`); on activate, calls `api.deleteBookmark(id)` then `loadAndRender(root)` (re-fetches the list).

## Acceptance criteria

These map to US-004's EARS clauses.

1. **AC1 (happy path):** `DELETE /api/bookmarks/:id` for an existing id → `204` no body; subsequent `GET /api/bookmarks` does not include the row. In the UI, clicking the row's delete control results in a re-render whose DOM no longer contains the row's `[data-bookmark="<id>"]`.
2. **AC2 (re-add):** After deleting a row whose URL was `u`, `POST /api/bookmarks {url: u, title: 't'}` returns `201` (no `DUPLICATE_URL`); `GET` includes a new row with a new `id`.
3. **AC3 (idempotent):** `DELETE /api/bookmarks/:id` for a never-existed or already-deleted id → `204`; remaining rows are unchanged; no thrown error in route handler; no `console.error` from the error middleware.
4. **Negative (non-integer):** `DELETE /api/bookmarks/not-an-int` → `400` `{error: {code: 'BAD_ID', message}}`.

## Behaviour-level test sketch

- `test/repo.test.ts` (delete section):
  - `insert(...)` then `deleteById(id)` → `true`; `list()` empty.
  - `deleteById(99999)` on empty DB → `false`; no throw.
  - `insert(url='u')`; `deleteById(id)`; `insert(url='u')` again → succeeds (different `id`).
- `test/routes-delete.test.ts` (supertest):
  - Seed via repo; `DELETE /api/bookmarks/<id>` → `204` no body; `GET` returns one fewer row.
  - `DELETE /api/bookmarks/<id>` repeated → still `204`.
  - `DELETE /api/bookmarks/abc` → `400` with `code: 'BAD_ID'`.
  - After delete + same-URL POST → `201`.
- `test/client-render.test.ts` (extension):
  - `renderList(root, [b1, b2])` produces two `[data-delete-for]` controls (button, type=button, `aria-label` mentions the bookmark title).
  - Wire a stub `api.deleteBookmark` that resolves; dispatch a click on the control for `b1`; assert `api.deleteBookmark` called with `b1.id`; assert a follow-up `api.listBookmarks` was called; assert the re-rendered DOM no longer contains `b1`.

## Notes

- `aria-label` on the delete button supports keyboard-only users — the cheapest accessibility primitive for a destructive control, and Design's `client/render.ts` is the place for it.
- No confirm dialog. The user is single-trusted-local; delete-then-re-add is the recovery path (Q04 / US-004 AC2). Adding a confirm step is out of scope per Spec's anti-scope list.
- Delete control is a `<button type="button">`, not an `<a>`. The link semantics belong to T-006's anchor only.
