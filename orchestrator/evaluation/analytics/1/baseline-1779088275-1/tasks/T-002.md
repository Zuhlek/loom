# T-002 — List bookmarks end-to-end (empty + populated)

## Goal

Deliver US-002 end-to-end: the user opens `http://localhost:3000`, the page fetches `GET /api/bookmarks`, and either the populated list or the empty-state message renders. This task lands the full vertical stack the later behavior tasks will graft onto: `db.ts` factory, `repo.listAll()`, the `routes.ts` GET handler, the Express wiring, the client API client, the render module, and the `main.ts` page-load orchestration.

## Scope

### Server

- `src/server/db.ts`: `createDb(filePath?: string): Database`. Default path resolves to `./data/bookmarks.db` (created if missing). Runs the schema bootstrap from `design.md` (`CREATE TABLE IF NOT EXISTS bookmarks (...)` plus the `idx_bookmarks_created_at` index) on every open. Tests pass `:memory:`.
- `src/server/repo.ts`: `listAll(): Bookmark[]` ordered by `created_at DESC, id DESC`. `insert` and `deleteById` are declared in the interface but their bodies can be stubs (or full implementations) — T-003 / T-005 own their behavior tests. `DuplicateUrlError` is exported; the actual throwing logic can be deferred to T-003 (Build's call; preferred to land it here since the UNIQUE constraint is created by the schema bootstrap).
- `src/server/routes.ts`: `GET /api/bookmarks` returns `{ bookmarks: Bookmark[] }` (200). Replaces the 501-placeholder router from T-001.
- `src/server/index.ts`: replace the placeholder router mount with the real `routes.ts` mount.

### Frontend

- `src/client/api.ts`: `fetchBookmarks(): Promise<Bookmark[]>` per the design's API client interface; throws `ApiError` with `status: 0, code: "NETWORK"` on `fetch` rejection.
- `src/client/render.ts`: `renderList(parent, bookmarks)` and `renderEmptyState(parent)` per the design. Each list item is an `<li>` containing an `<a>` for the title (anchor `href` set to the bookmark URL; the `target="_blank"` / `rel` attributes are wired up in T-004) and a visible URL string. Pure DOM helpers — no `fetch`, no document-level mutation other than appending to the passed `parent`.
- `src/client/main.ts`: on `DOMContentLoaded`, fetch the bookmarks, then call `renderEmptyState(...)` or `renderList(...)` against the `#bookmarks` mount. Owns the module-level `bookmarks: Bookmark[]` state and exposes an internal `refresh()` function used by T-003 and T-005.
- `public/index.html`: confirm the `#bookmarks` mount node and the `#add-form` mount node exist; the form's actual submit handler lands in T-003.
- `public/style.css`: list and empty-state styling.

### Tests (Vitest)

- `tests/unit/repo.test.ts`:
  - against `:memory:`, `listAll()` returns `[]`.
  - after two inserts at distinct `created_at` (clock stubbed), `listAll()` returns the newer one first.
  - after two inserts at the same `created_at`, the higher `id` appears first.
- `tests/unit/render.test.ts` (`// @vitest-environment happy-dom`):
  - `renderList(parent, [])` against an empty parent followed by `renderEmptyState(parent)` produces the empty-state DOM.
  - `renderList(parent, [{ id, title, url, created_at }, ...])` produces one `<li>` per bookmark, each with the title text and the URL accessible.
- `tests/http/bookmarks.test.ts`:
  - `GET /api/bookmarks` against the empty in-memory DB returns 200 `{ bookmarks: [] }`.
  - after seeding two rows via the repo, `GET /api/bookmarks` returns them newest-first.

## Notes for Build

- `repo.insert` and `repo.deleteById` are declared by the interface; landing their implementations here is fine and keeps the SQL surface together, but the **behavior tests** for them belong in T-003 and T-005 respectively. Do not duplicate the dedupe or idempotent-delete assertions here.
- The list-render path is the single dependency line the three remaining tasks ride on; keep its API stable.
