# T-005 — Delete a bookmark end-to-end with idempotent no-op

## Goal

Deliver US-004 end-to-end: the user activates the delete control on a list row, the server removes the row, the UI refetches and the row disappears. If the row was already deleted (e.g. in another tab), the second delete is a 204 no-op success and the UI still drops the row.

## Scope

### Server

- `src/server/repo.ts`: `deleteById(id: number): boolean` runs `DELETE FROM bookmarks WHERE id = ?`. Returns `true` when `changes === 1`, `false` when `changes === 0` (already-deleted row). Throws on non-integer / negative `id` only if not caught earlier — the route is responsible for input validation.
- `src/server/routes.ts`: `DELETE /api/bookmarks/:id`. Parses `:id` as a positive integer; returns 400 `INVALID_ID` if parsing fails. Calls `repo.deleteById`; returns 204 on both `true` and `false` returns (idempotent no-op per US-004 AC-3).

### Frontend

- `src/client/api.ts`: `deleteBookmark(id: number): Promise<void>`. Resolves on 204 (regardless of whether the row existed). Rejects on 400 / 500 / network failure with `ApiError`.
- `src/client/render.ts`: extend `renderList` so each `<li>` carries a per-row delete `<button>` with an accessible label (`aria-label="Delete bookmark"` or visible text) and a `data-bookmark-id` attribute set to the row's id. The button is initially enabled; `main.ts` toggles its `disabled` attribute during the in-flight delete.
- `src/client/main.ts`: delegate `click` events on the list container to the delete button. On click: disable the clicked button, call `deleteBookmark(id)`, then call `refresh()`. On `ApiError`, re-enable the button and surface a list-level non-blocking notice ("Couldn't delete — try again"); the design's per-row state machine `idle → deleting → (removed on refetch)` applies.

### Tests (Vitest)

- `tests/unit/repo.test.ts` (extends prior task's file):
  - `deleteById(existingId)` returns `true` and removes the row.
  - `deleteById(nonExistentId)` returns `false`, leaves other rows untouched.
- `tests/unit/render.test.ts` (extends prior task's file, happy-dom):
  - Each rendered list item carries a delete button with `data-bookmark-id` matching the row's id.
  - Setting the button's `disabled` flag via main's handler does not throw and does not lose the data attribute.
- `tests/http/bookmarks.test.ts` (extends prior task's file):
  - `DELETE /api/bookmarks/:id` for a seeded row returns 204; subsequent `GET` does not include it.
  - `DELETE /api/bookmarks/9999` against an empty/post-deleted state returns 204 (idempotent).
  - `DELETE /api/bookmarks/abc` returns 400 `INVALID_ID`.
  - `DELETE /api/bookmarks/-1` returns 400 `INVALID_ID`.
- `tests/client/api.test.ts` (extends prior task's file):
  - With `fetch` mocked to return 204, `deleteBookmark(1)` resolves.
  - With `fetch` mocked to return 400 `INVALID_ID`, `deleteBookmark("bad" as any)` rejects with `ApiError` whose `code === "INVALID_ID"`.

## Notes for Build

- The 204 + idempotent contract is the load-bearing part of US-004 AC-3. Build MUST NOT introduce a 404 path for "row does not exist on delete" — `decisions.md` and the design pin the no-op success.
- The per-row delete control is event-delegated from the list container in `main.ts`. Keeping the listener at the container level avoids re-binding on every refetch.
