# T-004 — Delete a bookmark from list and DB

## Observable behaviour

Each bookmark row in the rendered list shows a delete control (a
button labelled "Delete"). Clicking it sends `DELETE
/api/bookmarks/:id`; the server removes the row from SQLite and
returns `204`; the client re-fetches the list and re-renders without
the deleted row. The deletion is durable (persists across process
restart — gated by T-005). Deleting an already-removed id returns
`404 not_found`; the client re-fetches the list anyway so the UI
self-heals. No confirmation modal (ADR-007).

## Stories satisfied

- **US-004** — Delete a bookmark.
  - AC1: activating delete removes the row from SQLite and from the
    rendered list.
  - AC2: deletion persists across restart. T-004 owns the
    structural part (DB row is gone); T-005's restart gate verifies
    end-to-end across a process recreation that includes a delete.

## Scope

In:

- Repository `delete(id: number): boolean` — runs `DELETE FROM
  bookmarks WHERE id = ?`; returns `info.changes > 0`.
- HTTP route `DELETE /api/bookmarks/:id`:
  - Parses `:id` as integer; non-integer or NaN → `400 invalid_input`
    with `field` omitted.
  - Calls `repo.delete(id)`. If `true` → `204` no body. If `false`
    → `404 { error: { code: 'not_found', message } }`.
- Client:
  - `render()` adds a `<button data-action="delete" data-id="...">Delete</button>`
    per row.
  - Delegated click handler on the list container: on
    `data-action="delete"` activation, calls `deleteBookmark(id)`
    (DELETE request), then `loadBookmarks()` to re-fetch.
  - On 404, still calls `loadBookmarks()` (the row was already gone;
    self-heal).
  - On network error, surfaces a banner above the list ("Could not
    delete. Reload the page.").

Out:

- The restart-persistence E2E gate (T-005).

## Test sketch (Vitest)

`tests/repository.delete.test.ts`:

- Inserting then `repo.delete(id)` returns `true`; subsequent
  `repo.list()` does not contain the row.
- `repo.delete(id)` for a non-existent id returns `false`.

`tests/routes.delete.test.ts`:

- After a successful `POST`, `DELETE /api/bookmarks/:id` with that id
  returns 204 and a follow-up `GET /api/bookmarks` returns
  `{ bookmarks: [] }` (when it was the only row).
- `DELETE` for a never-existing id returns 404 with body
  `{ error: { code: 'not_found', message } }`.
- `DELETE /api/bookmarks/not-a-number` returns 400.

`tests/client.delete.test.ts`:

- `render({ bookmarks: [{ id:7, ... }] })` includes a button with
  `data-action="delete"` and `data-id="7"`.
- The delegated click handler resolves a click on a child of the
  delete button (event bubbling) to the same id.

## Acceptance

- All three new test files pass under `npm test`.
- Pre-existing T-001 / T-002 / T-003 tests still pass.
- Manual `npm start` smoke: clicking Delete removes the row from the
  visible list and from a follow-up `GET /api/bookmarks`.

## Notes / constraints

- One-click delete; no confirmation modal (ADR-007).
- Re-fetch after mutation, not local splice (ADR-004).
- The delete button is inside the row but outside the anchor, so
  clicking the title or URL never triggers delete (event scoping
  matters; the test asserts the delegation target).
