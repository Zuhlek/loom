# T-005 — Restart persistence end-to-end gate

## Observable behaviour

With T-001..T-004 in place, this task adds the explicit acceptance
gate that proves user-visible persistence: bookmarks survive a process
restart and a delete operation persists across the same restart, in
the newest-first order.

## Stories satisfied

- **US-005** — Bookmarks survive server restart.
  - AC1: after stopping the server and running `npm start` again,
    every prior bookmark is rendered in the same newest-first order.
  - AC2: a single SQLite file on disk is the canonical store —
    asserted by reading the same `BOOKMARKS_DB_PATH` from two
    independently constructed app handles in the test.

## Single-layer justification

T-005 touches only the `tests` layer because it is a cross-cutting
acceptance gate that asserts behaviour produced by T-001..T-004. No
new product code is needed; the test simulates "restart" by tearing
down the first Express app + DB handle and constructing a fresh pair
against the same on-disk SQLite file. This is the cheapest faithful
representation of `npm start` → stop → `npm start` for an in-process
test. The plan flags this as the single permitted single-layer slice.

## Scope

In:

- `tests/persistence.restart.test.ts` (Vitest):
  - `beforeAll`: create temp directory, set `BOOKMARKS_DB_PATH` to
    a file path inside it.
  - Stage 1: build app1 via `createApp(repoFromPath(dbPath))`. POST
    three bookmarks with distinct titles and URLs. DELETE the middle
    one.
  - Tear down: close app1's DB handle (release file lock). Discard
    the in-memory `repo` reference.
  - Stage 2: build app2 against the SAME `dbPath`. `GET
    /api/bookmarks` returns exactly the two remaining bookmarks, in
    newest-first order, with the same ids and `createdAt`
    timestamps observed in stage 1.

Out:

- Any new product code. If the test fails, the fix lives in
  T-001..T-004; T-005 is the assertion, not the implementation.

## Test sketch (Vitest)

`tests/persistence.restart.test.ts`:

- Setup writes to a real on-disk SQLite file (NOT `:memory:`).
- Stage 1 saves three rows, deletes one.
- Stage 2 reopens the same file via a fresh `createApp` call.
- Stage 2 assertion: `bookmarks.length === 2`; ids and `createdAt`
  match the stage-1 survivors exactly; the order is newest-first.
- Negative control: if the test is pointed at a temp file that does
  NOT exist between stages (the test deletes it on purpose),
  expectations flip and stage 2 returns `[]` — confirms the
  persistence claim hinges on the file, not on some test-runtime
  cache. (Optional sub-test; recommended.)

## Acceptance

- `tests/persistence.restart.test.ts` passes under `npm test`.
- All prior tests still pass.
- The test does not leak temp files or open DB handles.

## Notes / constraints

- The test MUST use a temp file path, NOT `:memory:`, since the goal
  is to prove that the disk file is the canonical store (ADR-009 also
  endorses on-disk for at least one test).
- `better-sqlite3` releases the file when the `Database` instance is
  GC'd or `.close()` is called explicitly; the test calls `.close()`
  for determinism.
