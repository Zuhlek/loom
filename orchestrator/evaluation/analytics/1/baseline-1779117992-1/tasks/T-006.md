# T-006 — Open bookmark in new tab

## Goal

Deliver US-003 ("Open a saved bookmark in a new browser tab"). The
"open" affordance is a standard `<a>` element rendered inside each row
template, with `target="_blank"` and `rel="noopener noreferrer"`.

This task touches the web layer only — it has no server change.
Single-layer justification: opening a URL in a new tab is browser
behaviour governed by anchor attributes. There is no server contract to
extend; adding HTTP plumbing would be ceremony. Tests assert the
attributes (the contract) at the web layer.

## Scope

In:
- Extend `renderList` in `src/web/main.ts` to emit, per row, an `<a>`
  whose:
  - `href === bookmark.url`,
  - `target === '_blank'`,
  - `rel === 'noopener noreferrer'`,
  - visible label is the bookmark's `title`.
- `tests/web.test.ts` — assertions on the anchor attributes per row.

Out:
- Server changes (there are none).
- Delete affordance (T-007).

## Acceptance gates (behaviour-level)

US-003 AC 1, AC 2.

## Test sketch

Web layer (`tests/web.test.ts`):
- Render two rows. For each row, the open anchor element's
  `getAttribute('href')` equals the bookmark's URL, `target === '_blank'`
  and `rel === 'noopener noreferrer'`. (US-003 AC 1.)
- The rendered HTML contains no `window.location` assignment, no
  router code, and no `<a>` without `target=_blank` for bookmark
  URLs. (US-003 AC 2 — the list page cannot be navigated away from
  the original tab via these anchors.)
- A bookmark whose URL contains characters that need escaping
  (`https://x.test/?a=1&b="quoted"`) renders correctly: `href`
  contains the exact URL, no double-escaping in the rendered DOM.

## Notes

- The security envelope (`design.md § Constraints`) requires
  `rel="noopener noreferrer"` on every outbound link.
- Do NOT use `window.open` for the open affordance — a plain anchor
  is the contract.
