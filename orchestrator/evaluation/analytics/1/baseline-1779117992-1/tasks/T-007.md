# T-007 — Delete bookmark end-to-end

## Goal

Deliver US-004 ("Delete a bookmark I no longer want") as one vertical
slice: `DELETE /api/bookmarks/:id` on the server, click handler + 404
reconciliation on the client, no confirmation modal.

## Scope

In:
- `src/routes/bookmarks.ts` — add `DELETE /:id`:
  - Parse `:id` as a positive integer; reject non-numeric with `400
    invalid_input`, `field: 'id'`.
  - Call `deleteBookmark(db, id)`. On success, return `204` with empty
    body.
  - On `NotFoundError`, return `404 not_found`.
- `src/web/main.ts` — extend `renderList` to emit a delete affordance
  per row (a `<button>` with the bookmark id in `data-id`). Wire the
  click handler:
  - Single click → immediate `deleteBookmark(id)` request, no
    confirmation modal (US-004 AC 3).
  - On `204`, remove the row from the local list and re-render.
  - On `404`, call `fetchBookmarks()` and re-render with the
    authoritative list (US-004 AC 2).
  - On other non-2xx, show the top-of-page error banner; list
    unchanged.
- `tests/api.test.ts` — DELETE cases.
- `tests/web.test.ts` — delete-click cases.

Out:
- Save / list / open (T-004 / T-005 / T-006).

## Acceptance gates (behaviour-level)

US-004 AC 1, AC 2, AC 3.

## Test sketch

API layer (`tests/api.test.ts`):
- Seed one bookmark via the repo. `DELETE /api/bookmarks/<id>` →
  `204`; subsequent `GET` shows the row gone.
- `DELETE /api/bookmarks/99999` against an empty / mismatched DB →
  `404 not_found`, envelope shape preserved.
- `DELETE /api/bookmarks/abc` → `400 invalid_input`.

Web layer (`tests/web.test.ts`):
- Render two rows. Clicking the first row's delete button fires
  exactly one DELETE request with the matching `id`. After the
  stubbed `204`, the row disappears from the rendered list and the
  other row remains.
- Clicking delete on a stale row (server returns `404`) triggers a
  follow-up GET; assert via stubbed-fetch spy that the sequence is
  `DELETE → GET`, and the rendered list reflects the GET result.
- The DOM contains no confirmation modal between the click and the
  fetch call — a single click fires the request synchronously (US-004
  AC 3 — structural).

## Notes

- The DELETE route MUST be mounted under `/api/bookmarks/:id` so the
  same `routes/bookmarks.ts` router carries all three verbs (ADR-005:
  GET / POST / DELETE only; no PATCH route).
- The 404 → refetch path is the only place the client deliberately
  re-syncs from the server mid-session; document it in code.
