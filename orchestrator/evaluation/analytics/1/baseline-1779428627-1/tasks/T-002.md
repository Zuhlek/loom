# T-002 — SQLite repository + schema migration

## Why this task exists

Owns the persistence contract that the HTTP routes (T-004) consume.
Single-layer (persistence) is justified because the
`BookmarkRepository` interface is the seam between the route layer and
`better-sqlite3`; pulling it into a route task would entangle the
`SQLITE_CONSTRAINT_UNIQUE` → `DuplicateUrlError` translation with HTTP
status mapping (ADR-002).

## Behaviour delivered

- The `bookmarks` table and `idx_bookmarks_created_at` index are
  created idempotently on first DB open.
- `repo.list()` returns rows ordered `created_at DESC, id DESC`.
- `repo.create({ title, url })` persists a row with `id` and
  `created_at` server-assigned and returns the full `Bookmark`.
- `repo.create` re-throws `SQLITE_CONSTRAINT_UNIQUE` as
  `DuplicateUrlError`.
- `repo.delete(id)` returns `true` when a row matched, `false`
  otherwise.

## Acceptance criteria (story-anchored)

- US-001 AC1: persisted with server-assigned `id` and `created_at`.
- US-001 AC3: duplicate `url` surfaces as `DuplicateUrlError`.
- US-002 AC1: `list()` ordering is newest-first deterministic.
- US-004 AC1: `delete(id)` removes the row.
- US-004 AC3: `delete(id)` returns `false` when no row matched, enabling
  the route layer to map to `404`.

## Test sketch

`test/server/repository.test.ts` (Vitest, in-memory SQLite per test):

- `list()` on empty DB → `[]`.
- `create({ title, url })` → returned object has `id: number`,
  `created_at: number`, and `title` / `url` echo back; a subsequent
  `list()` finds it.
- `create` two distinct URLs, then `list()` returns them in
  reverse-insertion order.
- `create` two rows with identical `created_at` (mocked clock) — tie
  broken by `id DESC`.
- `create` with a URL already present throws `DuplicateUrlError`;
  `list()` still has only the original row.
- `delete(existingId)` → `true`; subsequent `list()` no longer contains
  the row.
- `delete(nonExistentId)` → `false`; `list()` unchanged.
