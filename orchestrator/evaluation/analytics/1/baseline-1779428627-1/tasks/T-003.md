# T-003 — Server validation rules + error taxonomy

## Why this task exists

Server-side validation is the authoritative gate per spec § Validation
envelope. Centralising it in `src/server/validation.ts` lets the route
layer (T-004) stay focused on HTTP shape. Single-layer is justified
because the contract (which error class corresponds to which field /
HTTP status) is owned exclusively by this module.

## Behaviour delivered

- `parseCreateInput(body: unknown): { title: string; url: string }`
  throws `ValidationError` with `field: "title"` or `field: "url"` on
  bad input; returns the trimmed inputs on success.
- Title rule: non-empty after trim, length ≤ 2048.
- URL rule: trims input; `new URL(value)` succeeds; `protocol` is
  `http:` or `https:`.
- `parseIdParam(raw: string): number` throws `ValidationError`
  (field undefined) when the path param is not a positive integer.

## Acceptance criteria (story-anchored)

- US-001 AC4: empty title → `ValidationError({ field: "title" })`;
  malformed URL or non-http(s) URL → `ValidationError({ field: "url" })`.
- US-004 AC3: non-integer or non-positive `:id` → `ValidationError`
  (mapped to 400 by route layer; precondition for the route's 404 path
  to be reachable only on well-formed but absent ids).

## Test sketch

`test/server/validation.test.ts` (Vitest):

- `parseCreateInput({ title: "ok", url: "https://x" })` returns the
  trimmed pair.
- Empty / whitespace-only title throws with `field: "title"`.
- 2049-char title throws with `field: "title"`.
- `url` of `not-a-url` throws with `field: "url"`.
- `url` with `javascript:` or `ftp:` scheme throws with `field: "url"`.
- Non-object body throws (no field).
- `parseIdParam("0")` and `parseIdParam("-1")` and `parseIdParam("abc")`
  throw; `parseIdParam("42")` returns `42`.
