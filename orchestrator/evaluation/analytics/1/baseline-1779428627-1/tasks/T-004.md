# T-004 â€” HTTP routes for list/create/delete

## Why this task exists

The HTTP layer is the contract the client (T-006/T-007/T-008) and the
out-of-band browser walkthrough both consume. It cuts vertically across
validation (T-003) and persistence (T-002), but does not own either â€”
the route handlers receive a `BookmarkRepository` by injection per
ADR-002.

## Behaviour delivered

- `createApp(repo: BookmarkRepository): express.Express` returns a
  configured Express app with JSON middleware (16 KiB cap),
  `/api/bookmarks` routes mounted, and the unified error middleware
  mapping `ValidationError | DuplicateUrlError | NotFoundError | SyntaxError`
  to `{ code, message, field? }` JSON bodies with the documented status
  codes.
- `GET /api/bookmarks` â†’ `200`, `Bookmark[]` newest-first.
- `POST /api/bookmarks` â†’ `201` with the created `Bookmark`; `400`
  `validation_error`; `409` `duplicate_url`.
- `DELETE /api/bookmarks/:id` â†’ `204` on success; `400`
  `validation_error` for malformed id; `404` `not_found` for unknown id.
- All responses set `Content-Type: application/json; charset=utf-8`
  (except `204`).

## Acceptance criteria (story-anchored)

- US-001 AC1: `POST` valid input returns `201` with `id` and `created_at`
  set by the server.
- US-001 AC3: `POST` with a duplicate URL returns `409` `duplicate_url`;
  the table is unchanged.
- US-001 AC4: `POST` with empty title or invalid URL returns `400`
  `validation_error` with the offending `field`.
- US-002 AC1: `GET` returns rows ordered `created_at DESC, id DESC`.
- US-004 AC1: `DELETE` for an existing id returns `204` and the row is
  gone from storage.
- US-004 AC3: `DELETE` for an unknown id returns `404` `not_found` and
  storage is unchanged.
