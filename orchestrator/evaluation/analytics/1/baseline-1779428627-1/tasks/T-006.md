# T-006 — Client API client + list rendering

## Why this task exists

The shared client foundation that T-007 (create form) and T-008
(delete) build on. Owns:
- `api.ts` — typed `fetch` wrapper for the three endpoints, mapping 4xx
  responses to a thrown `ApiError`.
- `render.ts` — pure functions that turn `Bookmark[]` into the DOM,
  including the `target="_blank" rel="noopener noreferrer"` contract.
- `main.ts` — entry point that loads the list on `DOMContentLoaded` and
  wires the rendered DOM. Form / delete handlers are added by T-007 /
  T-008 against the same `main.ts`.

## Behaviour delivered

- `listBookmarks()`, `createBookmark(input)`, `deleteBookmark(id)` —
  the three functions in `src/client/api.ts`. 4xx responses throw
  `ApiError({ status, code, message, field? })`.
- `renderList(parent: HTMLUListElement, bookmarks: Bookmark[])` —
  clears the parent and renders one `<li data-id="...">` per bookmark.
- Each `<li>` contains an `<a href={url} target="_blank"
  rel="noopener noreferrer">{title}</a>`, a secondary text node showing
  the URL, and a `<button data-action="delete">` (the delete handler is
  wired in T-008).
- `main.ts` calls `listBookmarks()` on load, toggles `#empty-state`
  based on the result, and renders into `#bookmark-list`.
- On `listBookmarks()` failure: `#form-error` shows
  "Could not load bookmarks."; the list stays empty.

## Acceptance criteria (story-anchored)

- US-002 AC1: rows are rendered in the order received from
  `GET /api/bookmarks` (server already orders newest-first).
- US-002 AC2: every rendered `<li>` shows the title as link text AND
  the URL on a secondary line.
- US-002 AC3: when `bookmarks.length === 0`, `#empty-state` is visible
  and `#bookmark-list` has no `<li>` children; otherwise `#empty-state`
  is hidden.
- US-003 AC1: every rendered `<a>` has `target="_blank"`.
- US-003 AC2: every rendered `<a>` has `rel="noopener noreferrer"`.

## Test sketch

`test/client/render.test.ts` (Vitest + jsdom):

- `renderList(ul, [])` clears the `<ul>` and leaves it with no
  children.
- `renderList(ul, [b1, b2])` produces two `<li data-id="...">`
  elements, in the array's order, each with the expected `<a>` and
  delete button.
- Every `<a>` has `target === "_blank"` and `rel === "noopener noreferrer"`.
- Every `<a>` has its `href` set to `bookmark.url`.
- The visible URL text content matches `bookmark.url`.
- Re-running `renderList` replaces the contents (no duplicate rows).

`test/client/api.test.ts` (Vitest with `vi.stubGlobal('fetch', ...)`):

- `listBookmarks()` returns the parsed JSON array on `200`.
- `createBookmark({title,url})` resolves with the parsed body on
  `201`.
- `createBookmark` rejects with `ApiError({ status: 409, code:
  "duplicate_url" })` when fetch resolves with that response.
- `createBookmark` rejects with `ApiError({ status: 400, code:
  "validation_error", field: "title" })` on the matching 400.
- `deleteBookmark(id)` resolves on `204`.
- `deleteBookmark(id)` rejects with `ApiError({ status: 404, code:
  "not_found" })` on 404 (T-008 catches and treats as success at the
  caller level).

## Notes

- `render.ts` uses `document.createElement` + `textContent` — never
  `innerHTML` with user input (design § Security).
- `main.ts` exports the bookmark array reference so T-007 / T-008 can
  mutate it; alternatively the array lives in a small `state.ts`. Build
  picks one when implementing T-007.
