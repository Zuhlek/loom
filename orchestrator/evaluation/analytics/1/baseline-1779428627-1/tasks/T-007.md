# T-007 — Client create form behaviour with inline errors

## Why this task exists

End-to-end slice of US-001 on the client side: form submit triggers
`createBookmark`, the new row prepends to the list, and `409` /
`validation_error` responses show inline messages in `#form-error`.

## Behaviour delivered

- `src/client/validation.ts` — duplicate of the server rules per
  ADR-006: non-empty trimmed title, `new URL(value)` succeeds, scheme
  in `{ http, https }`.
- Submit handler attached to `#bookmark-form` in `main.ts`:
  1. Reads `#bookmark-title` and `#bookmark-url`, trims them.
  2. Runs client validation; on failure, populates `#form-error` with
     the matching message and does NOT call the API.
  3. On valid input, disables `#bookmark-submit`, calls
     `createBookmark`.
  4. On `201`: prepends the new row's `<li>` to `#bookmark-list`,
     hides `#empty-state` if shown, clears the inputs, clears
     `#form-error`, re-enables submit.
  5. On `ApiError(409, duplicate_url)`: populates `#form-error` with
     "That URL is already saved.", leaves inputs intact, re-enables
     submit.
  6. On `ApiError(400, validation_error)`: populates `#form-error`
     with a per-field message, leaves inputs intact, re-enables
     submit.
- Typing in either input clears `#form-error`.

## Acceptance criteria (story-anchored)

- US-001 AC1: valid submit calls `POST /api/bookmarks` and persists
  via the server. (Asserted via the api spec stub.)
- US-001 AC2: on `201` the new row appears at the top of
  `#bookmark-list` without a full page reload (no `location.assign`,
  no `<form>` default submit — handler calls `event.preventDefault`).
- US-001 AC3: on `409` the form displays a duplicate-specific inline
  error and the row count in the list does not change.
- US-001 AC4: empty title or invalid URL is caught client-side and
  the API is not called; the appropriate inline error is rendered.

## Test sketch

`test/client/form.test.ts` (Vitest + jsdom; loads the HTML shell, stubs
`fetch`, wires `main.ts` against the parsed DOM):

- Submit with empty title → `#form-error` says title is required; no
  `fetch` call.
- Submit with `url: "not-a-url"` → `#form-error` says URL is invalid;
  no `fetch` call.
- Submit with `url: "javascript:alert(1)"` → URL error, no `fetch`
  call.
- Submit with valid input + stubbed `201` response → `#bookmark-list`
  now has the new `<li>` at the top, `#empty-state` is hidden, inputs
  are cleared.
- Stubbed `409 duplicate_url` → `#form-error` shows a duplicate
  message; the list is unchanged.
- Stubbed `400 validation_error, field: "url"` → `#form-error` shows a
  URL error message; the list is unchanged.
- Typing in `#bookmark-title` after an error clears `#form-error`.

`test/client/validation.client.test.ts` — mirror of T-003's server
validation tests (title required, URL must be `http:` / `https:`).

## Notes

- The handler MUST call `event.preventDefault()` first thing; the
  full-page-reload regression is a real risk because `<form>` defaults
  navigate. (US-001 AC2 enforces this.)
