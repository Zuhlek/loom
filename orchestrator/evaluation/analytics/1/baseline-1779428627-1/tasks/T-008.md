# T-008 — Client per-row delete behaviour

## Why this task exists

End-to-end slice of US-004 on the client side: clicking the per-row
delete control removes the row from storage and from the rendered list.

## Behaviour delivered

- A delegated click handler on `#bookmark-list` watches for
  `button[data-action="delete"]` clicks.
- On click:
  1. Reads the parent `<li data-id="...">`'s `id`.
  2. Disables the button (in-flight guard) while the `DELETE` is in
     flight.
  3. Calls `deleteBookmark(id)`.
  4. On success (`204`): removes the `<li>` from the DOM, removes the
     entry from the in-memory `bookmarks` array, and shows
     `#empty-state` if the list is now empty.
  5. On `ApiError(404, not_found)`: treats it as success — removes the
     `<li>` anyway, since the row is already gone server-side (design
     § Row state).
  6. On unexpected error: re-enables the button and surfaces a generic
     "Could not delete bookmark." message in `#form-error`.

## Acceptance criteria (story-anchored)

- US-004 AC1: clicking delete sends `DELETE /api/bookmarks/:id` and
  the server removes the row (verified server-side in T-004; the
  client-side assertion is that the network request is made with the
  right id).
- US-004 AC2: on `204`, the matching `<li>` is removed from
  `#bookmark-list` without a full page reload.
- US-004 AC3 (client adapter): on `404`, the client does not crash and
  still removes the `<li>` to align with eventual server truth.

## Test sketch

`test/client/delete.test.ts` (Vitest + jsdom; loads the HTML shell,
renders two bookmarks via `renderList`, stubs `fetch`, wires the
delegated handler from `main.ts`):

- Click `[data-action="delete"]` on the first row, stub returns `204`:
  - `fetch` is called with `DELETE /api/bookmarks/<id1>`.
  - `#bookmark-list` no longer contains the row with `data-id="<id1>"`.
  - The other row is still present.
- Click delete on the only remaining row, stub returns `204`:
  - `#bookmark-list` is empty; `#empty-state` is visible.
- Click delete, stub returns `404 not_found`:
  - The row is still removed from the DOM (404 treated as already-gone).
- Click delete, stub rejects with a generic error:
  - The button is re-enabled.
  - `#form-error` shows a generic delete-failure message.
  - The row is NOT removed.
- The delete button is disabled while its `DELETE` is in flight (use a
  controllable promise stub to assert the `disabled` attribute during
  the call).

## Notes

- Event delegation off `#bookmark-list` keeps the listener count at
  one regardless of row count, and works for rows added later via
  T-007.
