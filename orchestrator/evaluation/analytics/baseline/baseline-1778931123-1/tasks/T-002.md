# T-002 — SQLite schema, migration, and repository

## Goal

Implement the only module that touches `better-sqlite3`. Defines the
`Bookmark` type, the migration (idempotent `CREATE TABLE IF NOT EXISTS` +
UNIQUE index on `url` + index on `created_at DESC`), and the `BookmarkRepo`
surface (`list`, `getById`, `create`, `deleteById`) with typed errors
(`DuplicateUrlError`, `NotFoundError`). URL canonicalisation lives here.

## Scope

- `app/src/types.ts` — `Bookmark`, `CreateBookmarkInput` per `design.md`.
- `app/src/db.ts` —
  - `openDb(path): Database`
  - `migrate(db): void` (idempotent, runs `CREATE TABLE IF NOT EXISTS`,
    `CREATE UNIQUE INDEX IF NOT EXISTS bookmarks_url_uniq`,
    `CREATE INDEX IF NOT EXISTS bookmarks_created_at_idx`)
  - `DuplicateUrlError`, `NotFoundError` classes with `code` fields.
  - `makeRepo(db): BookmarkRepo` with prepared statements cached on the
    instance.
  - URL canonicalisation: `new URL(rawUrl)` → reject if parsing throws or
    protocol is not `http:`/`https:`; canonical form is `url.toString()`.
  - `create` catches `SqliteError` with `code === 'SQLITE_CONSTRAINT_UNIQUE'`
    and rethrows `DuplicateUrlError`.
  - `deleteById` throws `NotFoundError` when `info.changes === 0`.
- `app/test/db.test.ts` — unit tests against `:memory:`.

## Behaviour delivered

- The data substrate for US-001 (persist + duplicate enforcement),
  US-002 (ordering), US-004 (delete + not-found, re-add-after-delete).
- HTTP layer is out of scope here — exercised in T-003.

## Acceptance criteria (derived from satisfied stories)

- US-001 AC1: `create` returns a `Bookmark` with `id`, `created_at`
  populated by the server; the row is visible at the top of `list()`.
- US-001 AC2: `create` with a canonical URL that already exists throws
  `DuplicateUrlError`.
- US-001 AC3 (data side): invalid URL → `create` throws (validation
  surfaces in T-003's route layer; here we throw a typed error so the
  route can map to `400`).
- US-001 AC4 (data side): repo `create` throws on empty-after-trim title
  (or the route layer rejects before the repo is called; pick one and
  document — recommendation: route layer rejects, repo trusts inputs that
  match the contract).
- US-002 AC1: `list()` returns rows ordered `created_at DESC, id DESC`.
- US-004 AC1 (data side): `deleteById(id)` removes the row.
- US-004 AC2 (data side): `deleteById(missingId)` throws `NotFoundError`.
- US-004 AC3: after `deleteById`, `create({url: sameUrl, ...})` succeeds.

## Test sketches

- `migrate` is idempotent (call twice, no error, schema unchanged).
- Inserting two bookmarks then `list()` returns the newer one first.
- Canonicalisation: insert `HTTPS://Example.com` and later attempt
  `https://example.com/` — second insert throws `DuplicateUrlError`.
- Reject `ftp://x.example/` and unparseable input (typed error from repo
  or route layer — see decision above).
- `deleteById(1)` removes row 1; `list()` no longer contains it.
- `deleteById(9999)` throws `NotFoundError`.
- Delete-then-create-with-same-URL succeeds (covers US-004 AC3).

## Notes

- Prepared statements created once per repo instance, not per call.
- Repo methods are synchronous; no Promise wrapping.
- `created_at` is server-generated via `new Date().toISOString()`.
