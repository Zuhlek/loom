# T-003 — API router and buildApp wiring

## Goal

Implement the Express router that adapts HTTP to the repo, plus the
`buildApp(repo)` factory so integration tests can spin the API up in-process
without binding a port. Owns request validation, the error envelope, and
HTTP status code mapping. No `better-sqlite3` import here.

## Scope

- `app/src/routes/bookmarks.ts`:
  - `bookmarksRouter(repo): express.Router`.
  - `GET /api/bookmarks` → `200 Bookmark[]`.
  - `POST /api/bookmarks` → `201 Bookmark` on success, with the validation
    pipeline from `design.md ## State and error handling`:
    1. Body must be object, `url` and `title` strings → else `400 validation`.
    2. Trim title; empty → `400 validation { field: 'title' }`.
    3. Parse URL; failure or non-http(s) protocol → `400 validation { field: 'url' }`.
    4. Canonicalise to `url.toString()` (or delegate to repo — choose one
       location consistent with T-002).
    5. `repo.create(...)`; catch `DuplicateUrlError` → `409 duplicate_url { field: 'url' }`.
    6. Any other throw → `500 internal`.
  - `DELETE /api/bookmarks/:id`:
    1. Parse `:id` base-10; NaN or non-positive → `400 validation`.
    2. `repo.deleteById(id)`; catch `NotFoundError` → `404 not_found`.
    3. Else `204`.
- `app/src/server.ts`:
  - `buildApp(repo): express.Express` — mounts `express.json()` and the
    router under `/api`. Static handler is added in T-004.
  - `startServer` stays in T-004.
- `app/test/api.test.ts` — `supertest` against `buildApp(makeRepo(openDb(':memory:')))`.

## Behaviour delivered

- HTTP surface for US-001 (POST with validation + duplicate error),
  US-002 (GET ordered), US-004 (DELETE with 404). Static UI and listen
  socket are out of scope here.

## Acceptance criteria (derived from satisfied stories)

- US-001 AC1: `POST /api/bookmarks` with valid `{url, title}` returns
  `201` and a `Bookmark` payload; subsequent `GET` lists it first.
- US-001 AC2: `POST` of an already-present URL returns `409` with
  `error.code === 'duplicate_url'` and `error.field === 'url'`.
- US-001 AC3: missing / empty / unparseable URL → `400` with
  `error.code === 'validation'` and `error.field === 'url'`.
- US-001 AC4: empty or whitespace-only title → `400 validation { field: 'title' }`.
- US-002 AC1: `GET /api/bookmarks` returns rows ordered
  `created_at DESC, id DESC`.
- US-004 AC1: `DELETE /api/bookmarks/:id` returns `204` and the row is gone
  from a subsequent `GET`.
- US-004 AC2: `DELETE /api/bookmarks/9999` returns `404` with
  `error.code === 'not_found'`; a `GET` afterwards is unchanged.
- US-004 AC3: `POST` succeeds with the same URL after the prior bookmark
  was deleted.

## Test sketches

- `POST` valid → `201`, body has `id` and `created_at`.
- `POST` valid then `GET` → list[0].url equals canonical form, list[0].id
  equals POST's response id.
- `POST` duplicate (after canonicalisation) → `409 duplicate_url`.
- `POST` invalid URLs (empty string, `not-a-url`, `ftp://x`) → `400 validation { field: 'url' }`.
- `POST` empty/whitespace title → `400 validation { field: 'title' }`.
- `POST` non-object body → `400 validation`.
- `DELETE /api/bookmarks/<existing>` → `204`; `GET` shows it gone.
- `DELETE /api/bookmarks/9999` → `404 not_found`.
- `DELETE /api/bookmarks/abc` → `400 validation`.
- No `Access-Control-*` headers in any response (single-origin constraint).

## Notes

- Error envelope shape exactly as in `design.md`:
  `{ error: { code, message, field? } }`.
- Router is pure: takes a repo, returns a Router. No global state, no env
  reads. Trivially testable.
