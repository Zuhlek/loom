# T-008 — In-row two-step delete confirmation

## Goal

Implement ADR-006: clicking a row's Delete button toggles its label to
`Confirm delete?` for ~5 seconds. A second click within that window calls
`deleteBookmark(id)` and refetches the list. Clicking elsewhere or letting
the timer elapse reverts to the idle label. After deletion, the row no
longer appears in the rendered list.

## Scope

- `app/src/client/delete.ts`:
  - `attachDeleteHandler(listRoot, refresh): void`.
  - Listens for clicks on elements matching `[data-action="delete"]` inside
    `listRoot`.
  - First click: store `pending = {id, timeoutId}`, swap button text to
    `'Confirm delete?'`, schedule `setTimeout(revert, 5000)`.
  - Second click on the same row (within timeout): call
    `deleteBookmark(id)`, clear timeout, await `refresh()`.
  - Click on a different row's delete button or anywhere outside the
    pending row's button: revert the pending row.
  - On `deleteBookmark` throwing (e.g. `404`): clear pending; render an
    error message in the form-error slot; still refetch (the bookmark may
    already be gone server-side).
- `app/src/client/main.ts`: wire `attachDeleteHandler` after bootstrap.
- `app/test/client-delete.test.ts` — jsdom tests with `fetch` stubbed and
  `vi.useFakeTimers()` for the 5-second window.

## Behaviour delivered

- US-004 AC1 fully at the client edge.
- US-004 AC2 surfaces as an error in the UI when the server returns
  `404 not_found`; the list remains unchanged after refetch.
- US-004 AC3 falls out automatically — after delete + refetch, the URL is
  free; a subsequent save with the same URL succeeds (verified end-to-end
  in T-007/T-009).

## Acceptance criteria (derived from satisfied stories)

- US-004 AC1: a confirmed delete results in `DELETE /api/bookmarks/:id`
  being called and the row being absent from the next render.
- US-004 AC2 (UI surface): when the API returns `404 not_found`, the UI
  surfaces an error and the rendered list is identical to the pre-click
  list (after refetch).
- US-004 AC3 (UI surface): after a delete, saving the same URL via the
  form succeeds. (Direct integration assertion is in T-007/T-009.)

## Test sketches

- First click swaps button text to `'Confirm delete?'`; second click
  within 5 s calls `fetch(DELETE /api/bookmarks/<id>)` and triggers
  refresh.
- First click then waiting 5 s (advance fake timer) → button text reverts
  to `'Delete'`; no `fetch` is made.
- First click on row A, then click on row B's delete button → row A
  reverts; row B becomes pending.
- Stub `fetch` to return `404`; first click + confirm → error message
  rendered; `refresh()` still called.
- Multiple confirms across the same row are idempotent: only one DELETE
  is sent per pending sequence.

## Notes

- Use event delegation on the list root, not per-row handlers (the list
  re-renders fully on each refresh — see ADR-005).
- The timer ID is stored on a module-local `pending` variable; reset on
  every revert.
