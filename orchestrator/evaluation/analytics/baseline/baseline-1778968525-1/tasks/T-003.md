## Goal

Ship the data-access layer over `better-sqlite3` with the three functions
the routes will consume. The repository owns all SQL and is the only place
that catches SQLite constraint errors and re-throws as the typed domain
error.

## Scope

- `src/server/repository/bookmarks.ts`:
  - Types: `BookmarkRow { id, title, url, created_at }`,
    `NewBookmarkInput { title, url }`.
  - `listBookmarks(db): BookmarkRow[]` — `SELECT ... ORDER BY created_at DESC, id DESC`.
  - `createBookmark(db, input): BookmarkRow` — parameterised insert;
    catches `SQLITE_CONSTRAINT_UNIQUE` and throws `DuplicateUrlError`.
  - `deleteBookmark(db, id): boolean` — parameterised delete; returns
    `true` if `changes > 0`, else `false`.
- `src/server/errors.ts` (initial sketch — finalised in T-005):
  - `DuplicateUrlError` class with `code = 'DUPLICATE_URL'`.
  - Stubs for `ValidationError`, `NotFoundError` (filled out in T-004,
    T-005 respectively) — or these can be added in their owning tasks.

## Acceptance criteria delivered

- US-001 AC1: `createBookmark` persists a row and returns it.
- US-001 AC4: `createBookmark` rejects duplicate URL via
  `DuplicateUrlError`.
- US-002 AC1: `listBookmarks` returns rows in newest-first order.
- US-002 AC4: `listBookmarks` returns `[]` when no rows exist.
- US-004 AC1: `deleteBookmark` removes a row from SQLite.
