## Goal

Wire the three HTTP routes that satisfy the canonical save/list/delete
operations, translating between repository calls and the API contract in
`design.md` §Interfaces.

## Scope

- `src/server/routes/bookmarks.ts`:
  - `GET /api/bookmarks` — calls `listBookmarks(db)`, maps each row to
    the API shape (`createdAt` camelCase), returns `{ bookmarks: [...] }`.
  - `POST /api/bookmarks` — runs `validateNewBookmark(req.body)`, calls
    `createBookmark(db, input)`, returns 201 with `{ bookmark }`. Errors
    propagated via `next(err)` so the global handler maps them.
  - `DELETE /api/bookmarks/:id` — parse `id` (positive integer; throw
    `ValidationError('INVALID_ID')` otherwise), call `deleteBookmark`;
    if `false` → `next(new NotFoundError())`; else `res.status(204).end()`.
- Update `buildApp` in `src/server/app.ts` to mount this router.

## Acceptance criteria delivered

- US-001 AC1 — POST persists and returns the new bookmark.
- US-001 AC2 — POST with invalid URL → 400 `INVALID_URL`, no write.
- US-001 AC3 — POST with empty title → 400 `INVALID_TITLE`, no write.
- US-001 AC4 — POST with duplicate URL → 409 `DUPLICATE_URL`, existing
  row unchanged.
- US-002 AC1 — GET returns newest-first.
- US-002 AC2 — Each entry carries title and url.
- US-002 AC4 — GET on empty DB returns `{ bookmarks: [] }`.
- US-004 AC1 — DELETE removes the row from SQLite.
- US-004 AC2 — DELETE on missing id returns 404 (UI surfaces the inline
  error in T-009).
