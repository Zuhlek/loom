# T-005: Delete bookmark slice

## Objective

Vertical slice that makes `DELETE /api/bookmarks/:id` remove the row,
respond `204`, return `404` for absent ids, and `400` for malformed
ids. Depends on T-002 for storage and the `parseId` helper landed by
T-004 (re-used; T-005 can declare an extra dep on T-004 if Build
sequences them, but `parseId` is small enough that T-005 may inline it
if dispatched first — see Notes).

## Deliverables

- `src/server/repo/bookmarks.ts` extended with:
  - `NotFoundError extends Error { id: number }`.
  - `deleteBookmark(db, id)`: prepared `DELETE`. If `changes === 0`,
    throw `NotFoundError`.
- `src/server/routes/bookmarks.ts` extended with `DELETE /:id`:
  - Parses `:id` via `parseId`; on failure → `400 { error: { code:
    "invalid_id" } }`.
  - Catches `NotFoundError` → `404 { error: { code: "not_found", id }
    }`.
  - Success → `204` empty body.

## Acceptance criteria covered

- **US-004 AC-1** — successful delete removes the row from SQLite;
  next `GET` does not include it.
- **US-004 AC-2** — deleting an absent id returns `404` (the inline
  error UI is owned by T-009).

## Test sketch

### Unit (`tests/unit/repo.test.ts` — extended)

- `deleteBookmark(db, knownId)` returns void; subsequent
  `listBookmarks` does not contain that row.
- `deleteBookmark(db, 999999)` on empty DB throws `NotFoundError`
  whose `.id === 999999`.

### Integration (`tests/integration/bookmarks.api.test.ts` — extended)

- Seed a row, `DELETE /api/bookmarks/:id` → `204`; subsequent `GET`
  returns the row absent.
- `DELETE /api/bookmarks/9999` against fresh DB → `404 { error: {
  code: "not_found", id: 9999 } }`.
- `DELETE /api/bookmarks/abc` → `400 { error: { code: "invalid_id" } }`.
- `DELETE /api/bookmarks/0` → `400`.
- `DELETE /api/bookmarks/-1` → `400` (likely caught by Express path
  pattern as 404; document the behaviour either way).

## Notes

- `parseId` is owned by T-004 (`validate.ts`). If Build dispatches
  T-005 before T-004, it should add T-004 to `blocked-by`. As written,
  both depend only on T-002 because they each own a single endpoint
  and can be developed in parallel against the same shared
  `validate.ts` file; the merge-time owner of `parseId` is whichever
  lands first.
