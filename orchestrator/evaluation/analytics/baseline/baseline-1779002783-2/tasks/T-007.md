# T-007: Client create-form flow

## Objective

Wire the `#new` form to `createBookmark`, re-fetch the list on
success, and surface inline errors (duplicate, invalid URL, invalid
title) into `#error` per US-001 AC-2/AC-3/AC-4. After every successful
POST, re-fetch and re-render (ADR-006).

## Deliverables

- `src/client/main.ts` extended:
  - Listens to `form#new` submit; prevents default.
  - Reads `url`, `title` input values.
  - Calls `api.createBookmark`; on success: clears `#error`, resets
    form fields, calls `fetchBookmarks` + `renderList`.
  - On `ApiError`: maps `code` to a user-facing message:
    - `duplicate_url` → `"That URL is already saved: ${url}"`.
    - `invalid_input` + `field === "url"` → `"That URL doesn't look
      right. Use http:// or https://."`.
    - `invalid_input` + `field === "title"` → `"Title can't be empty."`.
    - other → `"Couldn't save: ${message}"`.
  - Optional: client-side pre-check that mirrors `validateUrl`/`validateTitle`
    to short-circuit the round-trip (UX nicety, not security).

## Acceptance criteria covered

- **US-001 AC-2** — UI surface for duplicate `url` (server side owned
  by T-004).
- **US-001 AC-3** — UI surface for invalid URL.
- **US-001 AC-4** — UI surface for empty/whitespace title.

## Test sketch (`tests/unit/client/form.test.ts`)

- Under `happy-dom`: render the shell, dispatch a `submit` event with
  `url`/`title` populated; mock `fetch` to return `201`. Assert
  `#error` is hidden, form is reset, and `fetchBookmarks` is called
  exactly once after.
- Mock `fetch` to return `409 { error: { code: "duplicate_url", url
  } }`; assert `#error` is visible and its `textContent` includes the
  URL.
- Same shape for `400 invalid_input field=url` and
  `400 invalid_input field=title`.
- Network failure (fetch rejects) → `#error` shows a generic message.
