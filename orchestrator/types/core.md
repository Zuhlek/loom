# core Type

Patterns for the cinnamon investment platform ‚Äî backend (Node.js/Express) and frontend (React/Vite/MUI).

## Architecture

- **Monorepo:** pnpm workspace with 30+ packages (`cinnamon-*`), root `pnpm-workspace.yaml`
- **Backend:** `cinnamon-core` ‚Äî TypeScript, Express.js, PostgreSQL, Redis caching
- **Frontend:** `cinnamon-core-ui` ‚Äî React 18, Vite, Material-UI, Zustand, TanStack React Query
- **Shared types:** `cinnamon-api` (common), `cinnamon-core-api` (core domain interfaces)
- **UI infra:** `cinnamon-ui-base` (controllers, session state), `cinnamon-ui-library` (shared components, Storybook)
- **Server infra:** `cinnamon-server-base` (DB abstraction, Express helpers)

## Package Dependency Flow

```
cinnamon-api / cinnamon-base (shared, no framework deps)
  ‚Üí cinnamon-core-api (domain interfaces: IConsolidation, IBookingCode, etc.)
  ‚Üí cinnamon-server-base (DB abstraction, Express)
    ‚Üí cinnamon-core (business logic, API services, DB layer)
  ‚Üí cinnamon-ui-base (controllers, session state)
    ‚Üí cinnamon-ui-library (reusable React components)
      ‚Üí cinnamon-core-ui (pages, frontend services)
```

## Domain Model Patterns

- **Base class:** `IdentifiedObject { key: number; id: string; toJSON(); fromJSON(); copyFrom() }`
- **Enumerations:** `EnumerationObject` with `static set: IdentifiedObjectMap<T>` for static data (BookingCode, Currency, TransactionType)
- **JSON serialization:** Use `stringFromJson()`, `numberFromJson()`, `dateFromJson()`, `enumFromJson()`, `arrayFromJson()` from `cinnamon-api`
- **Interface naming:** `I` prefix ‚Äî `IConsolidation`, `IBookingCode`, `ITranslations`
- **Interfaces live in `-api` packages**, concrete classes in `-core`

## Key Domain Entities

- **InvestmentStructure:** Mandate ‚Üí Portfolio ‚Üí Position (SecurityPosition, CurrencyForward)
- **FinancialInstruments:** Security (Bond, equity), Currency, Issuer, MarketDataProvider
- **Consolidation:** Central reporting entity ‚Äî consolidates mandates/portfolios. Has `bookUntil`, `valuatedUntil`, `valuatedOkUntil` dates.
- **Booking/Accounting:** Transaction, BookingCode, ChartOfAccounts, Closing, ReturnType
- **Calculation engines:** Booker, Valuator, ACICalculator, BondFiguresCalculator, TargetStateBuilder, AutoReconciler

## Database Layer

- **Driver:** `pg` (node-postgres). `AbstractDatabase` uses `pg.Pool`/`pg.PoolClient`.
- **Pattern:** `DBClient { conn: ISDBConnection }` ‚Äî all DB-accessing classes extend this
- **Connection:** `ISDBConnection` wraps PostgreSQL, provides domain methods (`loadCurrencies()`, `getResolvedConsolidation()`, `loadEnumeration<T>()`)
- **Facade:** `ISDatabase` holds global caches (`securityCache`, `coaCache`, `userCache`, etc.) and version counters
- **Caching:** `IdentifiedObjectMap<T>` maps by numeric key; `TransientDataStore` for calculated data (histories); `StateInvalidator` for cache busting
- **Writes:** Use `batchOperation()` for bulk inserts. Wrap in `beginTransaction()`/`commit()`/`rollback()`
- **SQL files:** Raw SQL in `cinnamon-core/src/Database/sql/`

### PostgreSQL-specific patterns

- **Query results:** `conn.query()` returns `result.rows` (array of row objects). Column names are **lowercase** (PostgreSQL lowercases unquoted identifiers).
- **Placeholders:** Use `$1, $2, ...` numbered placeholders (not `?`). Filter clauses need `$N` renumbering when combined (via `parameterOffset` in `getSQLClause`).
- **Transaction behavior:** PostgreSQL aborts a transaction on any error ‚Äî subsequent commands fail with "current transaction is aborted". Error recovery must happen in a separate transaction.
- **Error codes:** `23505` = unique_violation, `23503` = foreign_key_violation, `40P01` = deadlock. Mapped in `mapSqlError()` / `signalsDBDeadlock()`.
- **Reserved words:** `User` is reserved ‚Äî table is named `User_`. Avoid other reserved words as table names.
- **Identity columns:** Use `GENERATED BY DEFAULT AS IDENTITY`. Sequences must be resynced after explicit key inserts (`resyncTableSequence()`).
- **Batch inserts:** `doBatchReturning()` builds multi-row `INSERT ... VALUES (...),(...) RETURNING key`. Query must include a `RETURNING` clause.
- **Text matching:** `POSITION($1 IN LOWER(field::TEXT)) > 0` for case-insensitive substring search. The `::TEXT` cast is needed for ENUM columns.
- **JSON:** Use `JSONB` columns. Operators: `->` (jsonb), `->>` (text), `?` (contains), `?|` (overlap). `COALESCE` replaces `IFNULL`, `jsonb_array_length` replaces `JSON_LENGTH`.

## Backend API Services

- **Framework:** Express.js with `@cinnamon/typescript-rest` decorators (`@Path`, `@GET`, `@POST`, `@PUT`, `@DELETE`, `@QueryParam`, `@PathParam`)
- **Base class:** `AbstractObjectService` ‚Äî provides `getDBConnection()`, `sendStateChangeRequest()`, `sendInsertRequest()`, `sendDeleteRequest()`, `handleError()`
- **Registration:** Services listed in `WebServer` constructor (15 services: BookingService, FinancialInstrumentsService, etc.)
- **Authentication:** `@Authenticated` decorator, multiple strategies (HeaderField, ClientCertificate, Mixed, Dummy)
- **Error handling:** `ServiceError` with `{ errorId, class, description, details, method, url }`
- **State broadcasting:** Changes sent via `StreamsRegistry` message streams to all connected clients

## Frontend Patterns

- **Controller singleton:** `Controller.instance` provides all API services (`baseService`, `bookingService`, `structureService`, etc.) + `queryClient`
- **Service layer:** `AbstractService` with `get<T>()`, `post<T>()`, `put<T>()`, `delete<T>()` returning `QueryResult<T>` / `MutationResult<T>` (wraps React Query)
- **Routing:** `createBrowserRouter` with lazy loading via `Loadable(lazy(() => import('./Page')))`
- **State:** Zustand stores + TanStack React Query for server state
- **Components:** MUI v5, Data Grid Pro v6, X Date Pickers Pro v6
- **Localization:** `UIDictionary.ts` (large file, 63KB) ‚Äî all UI strings
- **Session:** `SessionState` with `useUISettings()`, `useFilters()` hooks
- **Filtering:** `QueryFilter` hooks + `useTableState` + `CTable` component pattern

## Build & Compile

## Codebase Conventions

- **Reference-object typing:** Model classes always reference other class instances, never raw IDs. E.g., `ConsolidationIssue.consolidation` is always a `Consolidation`, never `Consolidation | string`. Method parameters follow the same pattern ‚Äî pass the object, not the ID.
- **toJSON() delegation:** Use `IdentifiedObject.toJSON()` to serialize references via their IDs and produce ISO dates. Don't hand-roll JSON serialization when the base class handles it.
- **SQL column naming:** Omit the table-name prefix from columns. E.g., use `IssueKey` not `ConsolidationIssueKey`, following the pattern used in other tables.
- **Avoid unnecessary JOINs:** If a getter method exists for a related entity (e.g., `getConsolidation()`), use it rather than JOINing the table in a query.
- **Schema versioning:** When adding or modifying database tables, always increment the schema version number and provide an update script (in `Database/sql/`). Also add DROP TABLE entries to `Drop.sql`.
- **FilterBuilder pattern:** Query filtering uses `*FilterBuilder` classes that produce SQL WHERE clauses. New queryable entities should include a corresponding FilterBuilder.

## Frontend Editor Patterns

- **useEditor idGetter:** Must return `undefined` (not `"undefined"` string) when the object has no ID yet. `useEditor` uses truthiness of `idGetter(data)` to decide whether to navigate after mutations ‚Äî a truthy string like `"undefined"` triggers navigation on every mount, causing infinite remount loops.
- **Sub-object editors:** When an editor manages sub-objects within a parent (e.g., strategy definitions within a strategy), all CRUD mutations (create/update/delete definition) should use the parent's PUT endpoint. POST creates a new parent; DELETE removes the entire parent ‚Äî neither is correct for sub-object ops.
- **Debugging render loops:** If `useRef` values reset between renders, the component is remounting (key change or parent unmount), not re-rendering. Check the `key` prop and `useEditor`'s navigate effects first.

## Triage Bias

- Backend-only changes (new DB methods, calculation logic) ‚Üí standard track
- Full-stack features (model + DB + API + UI page) ‚Üí standard track with task breakdown by layer
- UI-only changes (new page, component tweaks) ‚Üí standard track
- Calculation engine changes (Booker, Valuator, ACICalculator) ‚Üí standard track, requires test coverage
- Cross-package refactors or new shared packages ‚Üí deep track
