§8 — Prompt-cache-aware dispatch

new

Two concatenated bands per dispatch — a stable head (body + signature, verbatim, placeholders kept literal) and a dynamic tail (one <system-reminder> block). The closing </system-reminder> is the cached-prefix boundary. Everything above it hits Anthropic's prompt cache.

TL;DR
  • The lever: byte-stable prefix across dispatches of the same callable → prompt-cache hits.
  • The economy: ~90 % discount on cached input tokens (5-minute TTL). On a 30-task build with many re-dispatches, the dispatch-side bill collapses to the tail.
  • The discipline: orchestrator never substitutes <project>/<phase>/<task> into the head, never wraps the body in boilerplate.
  • Failure mode if dropped: every dispatch pays full input freight; build cost scales with total head × N rather than head + (tail × N).

The trade

What it buys
  • Cache hits on every re-dispatch of the same callable — Build coordinator (re-dispatched many times) and per-task Builders (~30×) are warm after the first call.
  • One convention, two payoffs: body/signature split also powers §7's silent schema check.
  • Localised cache misses — a paraphrase or new placeholder breaks this callable only.
  • Wrapper drift is structurally impossible — no place for the orchestrator to insert "context patches".
What it costs
  • Authoring discipline: agents must resolve placeholders from the tail at runtime, not expect them inlined.
  • 5-minute TTL: idle gaps invalidate the cache. Long HITL pauses pay first-call freight again on resume.
  • Cache-bug surface: a single rogue substitution in the head silently disables caching for that callable until detected.
  • No per-invocation tweaks: "just this once, also pass X" isn't an option — extension must go through the tail or a new signature.

Why it's not in Forge

ForgeLoom
Per-call prompts assembled inline by the orchestrator; dynamic identifiers interleaved with body text. Two-band concatenation enforced; head is verbatim file content, tail is a single <system-reminder>.
Body and signature mixed in one file (or no explicit signature). Body and signature are separate files — concatenated only at dispatch time.
Placeholders pre-substituted, breaking byte-stability. Placeholders kept literal in the head; agent resolves from the tail.
Theory linkage
Direct Principle A. §5 compresses references inside artifacts; §8 compresses the dispatch itself. Together they cover both axes of token waste — re-quoting content within a phase, and re-serialising the same head across phases. The convention also enforces Principle B at the wire: the head is the typed contract, the tail is the typed identifiers.

Open questions for discussion

  1. Cache-warmth observability: today we trust the contract holds. Should we measure cache-hit rate (via post-hoc transcript-harvest) and alert on regressions?
  2. The 5-minute TTL: long HITL gates can blow the cache. Worth a "warmup ping" before re-dispatch, or accept the freight?
  3. Per-project constitution overrides: if we ever need a project-specific head modifier, where does it land without breaking the byte-stable prefix?

Legend & abbreviations

Dispatch
The orchestrator handing a Task to a fresh subagent.
Callable
A body+signature file pair (phase agent, quality-check agent, or build method).
Stable head
Body file + thematic break + signature file, verbatim, with placeholders kept literal.
Dynamic tail
A single <system-reminder> block with the substituted project/phase/task/date.
Cached-prefix boundary
The closing </system-reminder> line — everything above it is cacheable.
TTL
Time-to-live — Anthropic's prompt cache expires entries after 5 minutes of inactivity.
Placeholder tokens
Literal <project>, <phase>, <task> strings in the head; never substituted by the orchestrator.
RETURN block
The structured YAML a phase agent emits at end of turn (extracted from signature.md ## Returns ### Return block).
Principle A
Context economy.
Principle B
Build-system semantics.
