§10 — Opt-in Quality Check

new

At every gate except Review, the user can opt into a narrow audit pass before deciding whether to rerun the phase. Cheaper than a full rerun, structured enough to feed the rerun if one happens.

TL;DR
  • The lever: a separate quality-check.md agent per phase, scoped to that phase's failure modes.
  • The economy: a small probe prices the rerun decision. User picks Continue or Rerun with evidence, not vibes.
  • The discipline: always user-opt-in; never automatic. Loom does not auto-loop on phase output.
  • Failure mode if dropped: the gate becomes a blind binary — rational play is "accept" more often than the project warrants.

The trade

What it buys
  • Priced rerun decision — the user sees what would change before paying for the rerun.
  • Targeted rerun, not blank-slate — every blocker/major finding must be addressed before the agent returns.
  • Per-phase scope — Spec QC looks at intent ambiguity; Plan QC at DAG holes and harness fit; Build QC at evidence sufficiency.
  • Cache-warm by inheritance from §8 — repeated QC invocations hit the prompt cache.
  • User-driven, never automatic — bounded token budget per gate.
What it costs
  • Extra subagent per phase — body + signature pair to author and maintain (4 of 5 phases).
  • An extra dispatch when the user opts in, paid before any rerun cost.
  • Calibration work — a too-strict QC flags noise; too-permissive QC adds nothing.
  • Cognitive load at the gate — one extra option to evaluate per phase boundary.

Why it's not in Forge

ForgeLoom
One review pass at end of pipeline (Phase 5). Review at end and per-phase opt-in Quality Check at every other gate.
Gate decision is "approve" or "redo" — no probe in between. Gate decision is "continue / probe / rerun / go back" — four options, with the probe priced.
Reruns re-enter with prior artifacts only. Reruns re-enter with prior artifacts plus quality-review.md as additional context.
Theory linkage
Principle A applied to the rerun decision: the cheapest defensible signal that a rerun is needed is a narrow audit, not a full re-roll. Principle B at the contract: the QC signature names what the evaluator looks at, so its findings are machine-routable into the rerun's input context. Without the opt-in probe, the user faces a binary they cannot price.

Open questions for discussion

  1. Should QC findings be persisted across reruns (audit chain) or overwritten each time?
  2. Severity calibration: should QC use the same Blocker/Major/Minor/Note scale as Review, or its own scoped vocabulary?
  3. Worth a "QC budget" UX hint — e.g. estimated tokens — before the user opts in?
  4. Should a QC pass that finds zero issues auto-advance the user to "Continue", or always re-prompt?
