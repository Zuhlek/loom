§11 — Repository pre-flight + cached architectural digest

new

First Spec dispatch in a workspace runs Explore once and writes two artifacts: a cross-fabric architectural digest (cached at .loom/.cache/) and a seed-relevant slice (per-project). A sha256 manifest invalidates per-file. Subsequent fabrics — and other projects against the same repo — read by reference rather than re-exploring.

TL;DR
  • The lever: repo understanding is written once, cited many times, invalidated by file-level hash.
  • The economy: a second project against the same repo skips the costly first-pass Explore entirely.
  • The discipline: Foundation grilling MUST NOT ask the user for facts the digest already states.
  • Failure mode if dropped: every new project (and every Spec rerun) pays full freight for architectural understanding the prior runs already produced.

Two artifacts, two scopes

ArtifactLocationScopeLifecycle
repo-digest.md .loom/.cache/ Cross-fabric facts: stack, topology, chokepoints, "where X lives" Persists across projects; per-file sha256 invalidation
repo-digest.manifest.json .loom/.cache/ schema_version, git_head, per-file sha256 list Rewritten when digest changes; checked on every read
repo-context.md .loom/<project>/ Seed-relevant slice: what this seed touches Per-project; cites digest sections rather than restating

The trade

What it buys
  • Cross-fabric amortisation — second project against same repo pays for slice only.
  • Bounded staleness — sha256 manifest invalidates exactly the files that changed.
  • Foundation grilling stops asking redundant questions — user turns go to genuine ambiguity.
  • First-pass cost paid once per repo, not once per project.
  • Auditable — what the agent "knows" about the repo is on disk and inspectable.
What it costs
  • First-call cost is real — full Explore pass on a cold cache is the most expensive Spec dispatch.
  • Manifest discipline — every digest claim must trace to a file in tracked_files, or invalidation silently misses changes.
  • Staleness risk if HEAD moves locally but cache isn't checked — relies on the manifest's git_head check actually running.
  • Two artifacts to keep coherent — digest sections and context cross-references must stay in sync on partial rewrite.

Why it's not in Forge

ForgeLoom
No persistent repo cache — each project re-explores from cold. Digest persists in .loom/.cache/ across projects on the same repo.
Questions phase asks user for facts that are also readable from the repo. Foundation grilling forbidden from asking what the digest or context file already names.
No structural cache invalidation; staleness is a vibes call. Per-file sha256 manifest — invalidation is structural and partial (only changed files re-explored).
Theory linkage
Principle A across two axes at once: within a project, the digest shrinks every Foundation question's effective context; across projects, a second fabric skips first-pass exploration. The sha256 manifest is Principle B applied to a context cache — typed by the file set it depends on, so invalidation is structural rather than vibes-based.

Open questions for discussion

  1. What's the right schema_version bump policy when the digest's section structure changes?
  2. Should the digest live outside .loom/ (e.g. a team-shared location) so multiple checkouts of the same repo benefit?
  3. When the user runs /weave on a dirty working tree, do we invalidate against working-tree files or against HEAD?
  4. Is there a privacy/security risk in persisting architectural facts to disk for projects that touch sensitive code?

Fabric
A Loom project (one workspace under .loom/<project>/).

Spec's first sub-phase: gather context before any decision questions. See §2.
Explore
The read-only search subagent that produces the initial digest.
HEAD
The current git commit (git rev-parse HEAD).
tracked_files
Manifest field listing only the files the digest actually depends on — not the whole tree.
