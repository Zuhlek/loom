New Concepts for Discussion

3 new sections

Three architectural mechanisms newly documented in lifecycle-concepts-toc.md (§8, §10, §11). They are present in Loom but were absent from Forge. Each page is a one-screen executive summary with a diagram and the gain/loss trade. Open them side-by-side to discuss.

Why these three, why now

The original TOC documented the phase-level choices (Spec/Design split, Review, vertical slicing, etc.) and the cross-cutting contracts (IDs, append-only state, typed signatures, capability minimisation, /tune). It did not cover the mechanisms below — yet each is load-bearing in production and absent from Forge.

The three concepts

§8

Prompt-cache-aware dispatch

Two-band dispatch (stable head + dynamic tail) keeps the prompt prefix byte-stable across calls. The cached prefix is what makes a 30-task build affordable.

Principle A · context economynew
 §10

Opt-in Quality Check

A narrow per-phase audit the user can fire before spending a full rerun. Turns the gate from a blind binary into a priced decision.

Principles A + B · gate economicsnew
 §11

Repository pre-flight + digest cache

Architectural facts written once per repo, cited by reference, invalidated per-file by sha256. Foundation grilling stops asking what the repo already states.

Principles A + B · cross-project amortisationnew

How they relate to Loom's two principles

Principle A

Context economy

Each cognitive pass sees the minimum sufficient context. §8 collapses dispatch freight; §10 keeps reruns priced before they're paid; §11 amortises architectural understanding across projects.

Principle B

Build-system semantics

Phases ship as typed rules. §8 makes the typed head byte-stable (cacheable); §10 routes structured findings back into the rerun's input; §11 types its own cache by per-file sha256, so invalidation is structural.
