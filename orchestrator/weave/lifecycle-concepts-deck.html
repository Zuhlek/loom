Theory

Lifecycle As Contracts

A contracted lifecycle treats delivery as a chain of explicit agreements, not as one continuous agent context.

Contract

  • owned by one phase
  • read by downstream phases
  • stable enough to audit
  • cheap enough to rerun locally

Effect

  • less implicit state
  • clearer failure ownership
  • smaller repair surface
  • better traceability

Phase

Spec: Intent Model

Spec converts raw request language into stable user-facing obligations.

Owns

  • what is being built
  • who receives value
  • scope boundaries
  • acceptance criteria

Does Not Own

  • component structure
  • implementation order
  • test harness mechanics
  • code-level tradeoffs

Theoretical value: separate product ambiguity from technical ambiguity.

Phase

Design: Structural Model

Design maps obligations onto components, interfaces, data, state, and failure modes.

Inputs

  • stories
  • constraints
  • repo conventions
  • accepted decisions

Outputs

  • system shape
  • interfaces
  • data/state model
  • ADRs and rejected alternatives

Theoretical value: Design may interpret Spec, but it cannot silently rewrite it.

Phase

Plan: Executable Graph

Plan translates structure into a graph Build can execute without inventing the work order.

Graph Invariants

  • no dangling dependencies
  • no cycles
  • every story covered
  • verification environment declared

Task Invariants

  • stable T-NNN identity
  • vertical slice shape
  • AFK / HITL class
  • story references

Theoretical value: move sequencing intelligence out of Build and into a checkable artifact.

Phase

Build: Isolated Execution

Build should execute planned slices, not reinterpret the project.

Coordinator

  • reads board
  • dispatches ready tasks
  • moves cards
  • aggregates evidence

Worker

  • fresh context
  • one task scope
  • red -> green loop
  • done report

Theoretical value: failure and context debris stay local to the task.

Phase

Review: Independent Judgement

Review compares the built result against upstream contracts and produces owned findings.

Checks

  • intent satisfaction
  • design conformance
  • plan completion
  • test evidence

Finding Shape

  • severity
  • evidence
  • expected vs actual
  • owner phase

Theoretical value: review is not another test run; it is contract interpretation.

Gates are not just UI pauses. They are control points where token spend, scope movement, and downstream invalidation become explicit.

Quality checks are optional meta-passes: useful when uncertainty is high, wasteful when the phase contract is already obvious.
